# 优化方案：位置一致的 PDF 译文（坐标驱动重排/覆盖）

## 目标与边界

你希望的效果是：翻译时从 PDF 的文字层获取到信息，经过 AI 编辑翻译后，再“原路返回”写出一个**译文 PDF**，并尽量做到**英译后的新 PDF 在视觉排版上与原位置高度对应**。

需要先明确两个边界：

1. 你们当前主输出是 Markdown，管线里没有保留 “文本片段 -> PDF 坐标/字号/行基线”的映射，因此无法直接做到“写回文字层”。
2. 即便新增坐标映射，PDF 里“逐字替换原文本对象”通常很难泛化；更可行的路线是生成一个译文的**覆盖层**（overlay）或**重排层**（reflow），用 bbox 将译文画回到对应区域。

因此本方案文档将“位置一致”拆成可落地的工程路径，而不是追求像素级完全一致。

---

## 多种方案（从易到难）

### 方案 A（推荐）：bbox 映射 + 覆盖绘制（overlay reflow）

实现角度：

1. **提取坐标**：用 PyMuPDF `page.get_text("dict")` 获取 block/line/span，并拿到每个 span 或每个“逻辑行”的 `bbox`、字体大小等信息。
2. **构建可翻译单元**：把 span 按阅读顺序合成为“行/短串”作为翻译输入；每个单元携带一个稳定的 `line_id`，以便把译文回填到对应 bbox。
3. **AI 翻译（结构约束）**：要求模型以“与输入行数一一对应”的方式输出，避免引入额外段落或乱序。
4. **覆盖绘制**：在新 PDF（或在原 PDF 的页面之上）对每个 bbox 使用 `page.insert_textbox`/`insert_text` 绘制译文，配合自动换行与字号自适配。

优点：

- 工程可控：不需要深入修改 PDF 的内容流对象。
- 易迭代：可以逐步扩大支持范围（单栏 -> 多栏；普通文本 -> 特殊字符等）。
- 失败可回退：生成 overlay 版译文 PDF，而不是破坏原文件。

缺点：

- 原文文本层通常仍存在（覆盖后可能出现重影或复制仍是原文，取决于你是否删除/遮盖原文）。
- 英文长度变化会触发溢出，需要 bbox 内换行/缩放策略。

适用场景：

- 大多数“文字层正常”的学术论文（尤其单栏正文）。
- 目录等结构区域（可在后处理阶段做专门规范化）。

---

### 方案 B：基于 bbox 的“结构翻译 + 重排排版”

实现角度：

1. 用 bbox 作为“约束模板”（例如每行的 x/y/宽度），但不再要求把译文画回到同样的 span 上。
2. 用排版器（简化版）重新排版：基于行高、换行规则、字体度量，把译文重新流到同一列宽内。

优点：

- 更像“新排版译文 PDF”，视觉观感更自然。

缺点：

- 更难逼近原版逐元素对应。
- 需要更多排版规则（行高、段落间距、缩进、字体度量）。

适用场景：

- 你希望译文 PDF 更“干净”，而不是严格逐 bbox 覆盖。

---

### 方案 C：写回 PDF 文本对象（真正替换文字层）

实现角度：

1. 从 PDF 文本抽取中定位到原 span 对应的内容对象（font、matrix、content stream 片段）。
2. 把对应文本替换为译文，同时保持同一套文本矩阵/字号规则。

优点：

- 复制/选中译文可能更正确。
- 理论上最接近“原路返回”的定义。

缺点：

- PDF 内部对象结构复杂，跨文档差异大，泛化成本高。
- 容易引入渲染差异或 PDF 文件损坏风险。

适用场景：

- 你愿意把它当成长期研究项目，并只做某类 PDF（例如固定模板的期刊）。

---

### 方案 D（折中）：只对关键区域 overlay（目录/段落标题/正文主干）

实现角度：

- 只对特定区域启用 overlay（例如：检测到目录页、或检测到主干正文的普通文本行）。
- 其它区域（表格、公式、图注）先跳过或保留原文。

优点：

- 复杂度显著下降。
- 更快得到“用户能用”的结果。

缺点：

- 并非全局统一覆盖，遇到复杂版式仍会不完整。

适用场景：

- 你想尽快看到效果，再逐步扩大覆盖范围。

---

## 最小可行里程碑（建议）

从“方案 A + 方案 D 思路”落地最稳：

1. M0：单栏、横排文本、bbox 提取稳定的页面，按“逻辑行”翻译并覆盖绘制。
2. M1：在 bbox 内启用换行 + 自动缩小字号直到不溢出（设置上限，避免过小）。
3. M2：对目录页启用更强的目录规范化（避免点线/页码乱序造成译文溢出）。
4. M3：逐步扩展到多栏与少量旋转文字（旋转过大的先跳过）。
5. M4：对公式/特殊符号采用“保留原样/跳过翻译”的策略（或仅翻译周边说明）。

## 风险控制建议（避免大改翻车）

- 先做“新输出文件”而不是覆盖原 PDF：默认产物命名为 `translated_reflowed.pdf`。
- 先限制适用范围：仅处理“文字层充足 + 横排 + 非旋转”的页面。
- 每页保留回退：某页渲染失败则退回原页并记录日志，不中断整任务。
- 给 UI 增加实验标记：例如“实验功能：位置一致译文 PDF（Beta）”。

## 与现有代码的对接点（需要新增能力）

你们当前的抽取与翻译入口：

- 文本抽取与分块：`pdf_translate/pipeline.py`（`_page_rows_for_main` 使用 `page.get_text("text")`）
- 目录噪声处理：`pdf_translate/text_sanitize.py`（`collapse_toc_dot_leaders`）

为了实现 overlay reflow，需要新增一条“抽取坐标层”的管线：

- 在页面级别改用 `page.get_text("dict")`，并构建 `logical_lines: [{line_id, bbox, text}]`。
- 翻译时以 `logical_lines` 的 `text` 为输入，输出一一对应的译文字符串数组。
- 新增一个 `pdf_reflow.py`（或放入现有模块）来完成 overlay 绘制，输出 `output/translated_reflowed.pdf`。

## 评估指标

建议用可量化的方式评估每轮迭代：

- 溢出率：bbox 内是否频繁 overflow（可统计插入失败/超限次数）。
- 视觉一致度：人工抽样给分，或用简单像素差度量（先人工）。
- 字符完整性：译文是否漏字/乱序（对齐校验 line_id）。
- 性能：每页 overlay 的耗时；并联/批量绘制策略是否能接受。
