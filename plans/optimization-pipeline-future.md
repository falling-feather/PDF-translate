# 后期可行优化（扩展 backlog）

除对话中重点讨论的 OCR 与目录版式外，下列方向与当前架构兼容，可按价值与成本排队。

## 阅读顺序与多栏正文

- **问题**：非目录页的双栏论文，`get_text("text")` 也可能列间错乱。
- **思路**：与目录 bbox 合并类似，按块排序规则（如分栏检测、x/y 排序）生成单行流；可先对「正文字符少但块数多」的页面启用。

## 结构化输出与后处理

- 对表格、算法伪代码等，考虑 **轻量检测**（启发式或关键词）+ 提示模型使用固定 Markdown 围栏（如表格用 `|` 对齐），译后可用简单校验/回退。

## 链接与图片迁移（设计文档延续）

- `PROJECT_DESIGN.md` 已讨论超链接与图片策略；当前有 `rich_content` 链接元数据与 `export_links`。
- 后续可增强：**双语 Markdown 脚注/引用**、按页关联图片导出路径（需明确是否与翻译块对齐）。

## 记忆与术语

- 强化 `memory/` 注入策略：按页/章节过滤术语子集、Top-K 高频词，控制 token（设计文档已有方向）。
- 可选：从译后差异或用户修正学习写回 `glossary.json`（需 UI 或 CLI 工作流）。

## 运维与体验

- **上传体积与超时**：大 PDF + OCR + 长上下文时，调整 Web 与 httpx 超时、分块重试策略（`http_retry` 已存在）。
- **任务队列**：多用户时考虑独立 worker 进程，避免长任务阻塞事件循环（视 `jobs.py` 演进而定）。
- **可观测性**：结构化日志（job_id、页码、后端、token 估算）便于排查失败与费用。

## 安全与部署

- 生产环境：HTTPS、CORS 收紧、管理员密码与 `PDF_TRANSLATE_BOOTSTRAP_*` 流程复核（`DEPLOY_WINDOWS_SERVER.md` 已部分覆盖）。

## 不建议过早投入的方向

- 全文版面重建为「译版 PDF 与原版像素级一致」——成本高，可与「Markdown + 原图/链接索引」路线权衡后再定。
