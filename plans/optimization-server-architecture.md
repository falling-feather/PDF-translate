# 服务端优化清单（全盘复审）

本文基于当前服务端实现（FastAPI + SQLite + 线程任务）整理，目标是提升稳定性、可观测性与可维护性。

## P0（优先）

### 1) 任务状态双写 DB（不仅写磁盘）

- 现状：任务运行状态主要保存在 `web_status.json` 与进程内内存，`jobs_meta` 只有提交元数据。
- 问题：服务重启、前端刷新、后台查询时，列表层面的状态信息不足；若仅看 `jobs_meta`，无法直接知道 done/error/running。
- 建议：
  - 在 SQLite 增加 `jobs_state`（`job_id/status/phase/updated_at/duration/error`）。
  - `JobRegistry.update()` 时同步写入 DB（可做节流，避免过高写频率）。
  - 用户与管理端列表直接读 DB，详情页再读内存/磁盘补充细节。

### 2) 下载 Zip 改为流式返回

- 现状：`zip_job_outputs()` 返回 `bytes`，整个 zip 在内存中构建后一次性返回。
- 风险：大任务并发下载时内存峰值高。
- 建议：改为 `StreamingResponse` + 临时文件（或分块写流），下载完成后清理临时文件。

### 3) JWT 增加过期时间与刷新策略

- 现状：JWT 只含 `sub/username/role`，没有 `exp`，token 理论长期有效。
- 风险：泄漏后长期可用，不利于生产部署。
- 建议：
  - 增加 `exp/iat`（例如 12h）。
  - 可选增加 refresh token 或重新登录机制。

### P0 实施提示

- 若你希望先拿到“可见收益”，建议优先顺序：
  1. 状态双写 DB（列表可见）；
  2. Zip 流式下载（稳定性）；
  3. JWT 过期策略（安全）。

## P1（中期）

### 4) 任务保留策略（TTL）

- 目标：实现「如 24 小时保留」。
- 建议：
  - 新增环境变量：`PDF_TRANSLATE_JOB_RETENTION_HOURS`。
  - 清理器按策略删除 `web_jobs/<job_id>/`；DB 保留可选（保留元数据但标记 artifact 过期）。
  - 删除策略写入审计日志，便于排查用户“找不到文件”。

### 5) 后端设置一致性校验

- 现状：管理员可设置 `default_backend` 和 `enabled_backends`，但缺少强校验。
- 建议：保存设置时校验默认后端必须在启用列表中，否则拒绝保存或自动回退到首个启用项。
- 备注：默认值已切到 DeepSeek，但该“联动校验”仍建议保留，防止管理端误配置造成默认值失效。

### 6) 审计事件分页与筛选

- 现状：有 `limit/offset`，但缺少按 `action/job_id/username` 条件筛选。
- 建议：给管理端审计页添加筛选参数，后端 SQL 增加可选条件。

## P2（可选增强）

### 7) 任务执行器从线程迁移为队列 Worker

- 现状：每任务一个守护线程，简单直观。
- 风险：高并发时线程数量增长、资源管理粗糙。
- 建议：演进为受控 worker 池（或进程队列），可设置并发上限与排队策略。

### 8) 前后端进度通道从轮询改为 SSE/WebSocket

- 现状：前端轮询 `/api/jobs/{id}`。
- 优化：推送式进度更实时，减少请求量。

### 9) 关键路径指标化

- 建议记录每任务的结构化指标：拆分耗时、翻译耗时、块数、失败重试次数、下载次数等；可先落库后再考虑监控面板。
