# 优化方案：扫描版 / 图片型 PDF 与 OCR

## 目标

在「无可选中文本层」的 PDF 上仍能产生可翻译的正文，同时控制对 **费用、耗时、质量** 的负面影响（尤其对 DeepSeek 等按 token 计费的后端）。

## 原则

1. **默认不改变现有行为**：文字层充足的页面仍用 `get_text("text")`。
2. **OCR 作为分支**：仅在检测为「疑似扫描 / 纯图文字」时启用，或作为 Web/CLI **显式选项**。
3. **输出接现有流水线**：OCR 结果写入与当前相同的「按页字符串」结构，后续 `chunking`、记忆、翻译器 **无需重写**。

## 阶段 A：检测与产品提示（优先、低成本）

- **指标（可按页）**：
  - `get_text` 得到的可打印字符数（可设阈值，如 `< 50` 字符/页且页面有图像）。
  - 已有 `extract_page_rich_meta` 中的 `image_count` 等，可组合启发式。
- **行为**：
  - Web：上传后或任务开始前提示「当前 PDF 疑似扫描版，标准模式可能无法翻译；请使用 OCR 模式或先 OCR 再上传」。
  - CLI：非零退出或警告信息，避免静默空结果。

**可行性**：高；无重型依赖。

## 阶段 B：可选整页 OCR 管线

- **技术选型（需评估维护成本与许可证）**：
  - PyMuPDF 内置 OCR（若当前锁定版本支持且环境可装 Tesseract）。
  - 或独立调用 Tesseract / PaddleOCR：将页面渲染为图像（注意 DPI，如 200–300）再识别。
- **集成点**：在 `_page_rows_for_main`（或等价单一入口）中，对「启用 OCR 的页」用 OCR 文本替换 `get_text` 结果。
- **注意**：
  - **公式、上下标、希腊字母**：OCR 错误率高，与系统提示中「保留公式」冲突，需在文档与用户提示中 **降低预期**。
  - **双栏**：需阅读顺序策略（见 [optimization-toc-layout.md](./optimization-toc-layout.md) 中 bbox 思路，可复用同类逻辑）。
  - **耗时**：整页 OCR 显著增加总时间；并联翻译不能省掉 OCR 前置成本。

## 阶段 C：与 DeepSeek / OpenAI 兼容后端的衔接

- **不必改**：`OpenAICompatibleTranslator`、分块大小、`TranslationRequest` 结构。
- **会变**：
  - 扫描文档的 **API 调用量与输入 token**（从接近零到正常水平）。
  - 若全文强制 OCR：**全体用户** 的 token 与延迟可能上升。

## 验收建议

- 准备三类样例：纯文字层 arXiv、扫描版论文、图文混排（文字层+大图）。
- 对比：无 OCR / 按需 OCR / 强制 OCR 的 **字符数、任务耗时、DeepSeek 费用（估算）、人工可读性**。

## 依赖与运维

- OCR 引擎与语言包体积、服务器是否允许安装 Tesseract、是否走 Docker 镜像，需在 `SETUP_MANUAL.md` 或部署文档中 **单独一节** 说明（实现后再改主文档即可）。
