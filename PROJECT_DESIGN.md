# PDF 英文学术文献智能翻译工具 — 项目设计与工作计划

本文档基于你的需求整理：**整体目标**、**系统架构与记忆机制**、**分阶段工作计划**，并逐条回答你提出的三个问题。

---

## 一、需求摘要（你理想中的工作流）

| 环节 | 目标 |
|------|------|
| 全局审视 | 识别参考文献、附录、致谢等「相对低优先级」区块，拆成独立 PDF，默认不翻译，节省算力与费用。 |
| 价值内容 | 对正文等「高价值」部分按 **1–3 页** 为单元切分，便于在模型上下文窗口内连贯处理。 |
| 记忆文件夹 | **双源构建**：(1) 自动阅读生成的关键词、梗概、术语表；(2) 翻译过程中人工或半自动确认后的持续更新，形成可复用的项目记忆。 |
| 连贯与术语 | 通过记忆注入（术语表 + 前文摘要）缓解「分段翻译」导致的术语不一致与上下文断裂。 |

## 二、总体架构（建议）

```
PDF 输入
   │
   ▼
[版面与结构分析] ──► 目录/章节/参考文献启发式 + 可选 LLM 辅助分类
   │
   ├─► 低优先级区块 ──► 导出为 reference-appendix.pdf（可选：仅 OCR/不译）
   │
   ▼
[正文按页/块切分] ──► 1–3 页/块 + 块间重叠（如半页）保证衔接
   │
   ▼
[每块流水线]
   ├─ 读取 memory/（术语表、实体、前文摘要）
   ├─ 可选：抽取本块术语候选 → 写入待确认队列
   ├─ 翻译（LLM 或 学术 API 或混合）
   └─ 更新 memory/（摘要、新术语、用户修正）
   │
   ▼
[输出] ──► 译文 PDF/HTML/Markdown（见下文「超链接与图片」）
```

**技术栈方向（可迭代）：**

- **PDF 解析**：`pymupdf`（fitz）、`pdfplumber`；复杂双栏/公式多时可考虑 `marker`、`nougat` 等（按需引入，避免第一期过重）。
- **结构化拆分**：页码范围 + 章节标题正则 + 参考文献关键词（References、Bibliography 等）+ 可选一次「轻量 LLM」对目录做分类。
- **状态与记忆**：本地文件夹 `memory/`（JSON/YAML + 可选 SQLite）；版本控制建议对 `memory/` 单独 git 或导出备份。
- **编排**：CLI 优先（`typer`/`click`），后续可加简单 Web UI。

## 三、记忆文件夹（`memory/`）设计

建议拆成多个文件，便于程序读写与人工抽查：

| 文件 | 内容 | 更新方式 |
|------|------|----------|
| `glossary.json` | 术语：英文、首选译文、别名、首次出现页码、置信度 | 自动候选 + 用户确认；翻译后纠错写回 |
| `entities.json` | 人名、机构、数据集、模型名等 | 与 glossary 类似，可合并或分表 |
| `running_summary.md` 或 `chunk_summaries.json` | 每块 3–8 句摘要 + 「当前叙事线索」 | 每译完一块追加；下一块翻译时作为上下文注入 |
| `style_notes.yaml` | 人称、时态、是否保留公式编号风格等 | 用户配置 + 少量自动推断 |
| `pending_review.json` | 待确认的术语/拆分边界 | UI 或 CLI 交互后清空或归档 |

**注入策略（控制 token）：** 每次调用模型时只带「与当前块相关的术语子集」（按页码/章节过滤）+ 最近 N 块摘要 + 全局高频术语 Top-K。

## 四、对你三个问题的回答

### 问题 1：开发中如何适当调用 AI？API 还是更便捷方案？

**结论：以「可编程的 API（云端或本地）」为主流程最合适；「纯 GUI 聊天」只适合原型，不适合长文档流水线。**

| 方式 | 优点 | 缺点 |
|------|------|------|
| **官方 HTTP API**（OpenAI、Anthropic、国内合规大模型等） | 可脚本化、可重试、可计费监控、易接记忆注入 | 需密钥与网络；要注意速率与成本 |
| **本地模型 API**（Ollama、vLLM、LM Studio 等） | 数据不出站；适合敏感文献 | 需显卡或接受较慢速度；质量依赖模型选型 |
| **IDE/Copilot 类** | 对人写代码方便 | 无法稳定驱动「上百段」批量翻译与状态机 |
| **RAG 平台无代码** | 上手快 | 对「按页切分 + 记忆写回 + PDF 重组」往往要大量定制 |

**「适当调用」的实践建议：**

1. **抽象一层 `Translator` 接口**：实现类如 `OpenAIClient`、`AnthropicClient`、`OllamaClient`，业务只依赖接口，便于换模型或 A/B。
2. **分段 + 限流 + 断点续跑**：每块一个 id，结果落盘；失败可重试单块，避免全文重跑。
3. **两阶段可选**：先用小模型/规则做结构切分；翻译用强模型；降本时可对「纯图表页」跳过或简译。
4. **日志与可复现**：保存每次调用的模型名、温度、prompt 版本（可 hash），便于排查「为什么这句译坏了」。

### 问题 2：除了 AI 翻译，能否接入专业学术翻译 API 优化工作流？

**可以，而且推荐做成「可插拔策略」：术语与长句用学术服务，叙述段落用 LLM，或全文走学术 API 再用 LLM 润色。**

常见思路（名称随地区与版权政策变化，选型时以最新文档为准）：

| 类型 | 典型用途 |
|------|----------|
| **机器翻译 API**（如 DeepL、Google Cloud Translate、Azure Translator、亚马逊 Translate 等） | 批量、便宜、低延迟；对论文「平铺直叙」部分效果好 |
| **面向垂直领域的 MT** | 若供应商提供「学术/技术」模型或术语库，可优先用于定义、方法节 |
| **自建术语库 + MT** | 将你 `glossary.json` 导出为 CAT/MT 术语表格式（若 API 支持） |

**推荐工作流组合：**

1. **段落级路由**：标题、摘要、结论 → LLM（更重连贯与语气）；公式说明、方法步骤 → MT + 轻量 LLM 校对。
2. **LLM 只做「译后编辑」**：MT 出初稿 → LLM 在固定指令下统一术语（必须遵守 glossary）、合并拆句、检查指代。
3. **成本敏感时**：全文 MT 第一遍 → 仅对低置信度段落调用 LLM。

实现上仍为同一 `Translator` 抽象，增加 `DeepLTranslator`、`HybridTranslator` 即可。

### 问题 3：论文中的超链接和图片如何合理迁移？

**核心原则：翻译的是「文本层」；链接与图保持可定位、可对照，而不是强行塞进纯 TXT。**

**超链接**

- **提取**：从 PDF 注解/链接对象读出 URL 与矩形区域；若仅有脚注式 URL，用正则从文本提取。
- **输出策略**：
  - **HTML**：`<a href="...">` 保留；译文在旁或下方。
  - **双语 Markdown**：`[原文片段](url)` 或脚注形式。
  - **新 PDF**：难度最高——可用报告类生成（如先 HTML 再转 PDF），或用库在译文旁放置链接注解；简单方案是「译文 PDF + 单独链接清单 CSV」。

**图片**

- **类型区分**：矢量图、位图、屏幕截图、公式渲染图。
- **处理选项**（可配置）：
  1. **原图嵌入输出**：保持分辨率，翻译仅加中文图注（caption），正文引用「见图 n」。
  2. **OCR + 译图注**：对截图内英文用 OCR，图注译后写入输出；图内小字一般不译（成本高且易错）。
  3. **多模态模型**：对「必须读懂图中文字」的页，整页或裁剪区域送 VLM，生成中文说明，作为附录或边注（注意 token 与费用）。

**与「1–3 页单元」的配合**

- 每块元数据记录：`page_range`、`figure_ids`、`link_count`；输出时按块拼装，避免链接错位。
- 图表页可标记为「低文本密度」，允许更大页范围或跳过正文模型调用，仅处理 caption。

## 五、工作计划（分阶段）

### 阶段 A — 最小可行产品（MVP）

1. PDF → 按页文本抽取 + 简单参考文献检测（关键词 + 最后 N 页启发式）。
2. 导出 `references.pdf` 与 `main.pdf`（页码集合）。
3. 正文按 1–3 页切块 + 可选重叠；单块调用一个 LLM API 翻译；结果合并为 Markdown。
4. `memory/glossary.json` + `running_summary` 手动/半自动更新（脚本追加）。

### 阶段 B — 记忆与质量

1. 每块译前自动检索相关术语；译后自动抽取候选术语进 `pending_review.json`。
2. 混合翻译：`HybridTranslator`（MT + LLM 校对）。
3. 断点续跑、配置项（模型、块大小、是否译参考文献）。

### 阶段 C — 版式与富内容

1. 图与链接元数据导出；HTML 双语对照视图。
2. 可选：VLM 对复杂插图页生成说明；公式友好导出（LaTeX 或保留为图片）。

### 阶段 D — 产品化（按需）

1. 简单桌面或 Web UI；项目管理（多篇论文多个 `memory/`）。
2. 隐私模式：默认本地模型；云端仅可选。

## 六、风险与对策（简要）

| 风险 | 对策 |
|------|------|
| PDF 结构混乱（双栏、脚注） | 优先 pymupdf 阅读顺序；差时换工具或人工标页范围 |
| 术语漂移 | 强制 glossary 注入 + 译后一致性检查 pass |
| API 费用 | 块大小动态调整；低价值页跳过；MT 打底 |
| 版权与合规 | 仅处理你有权使用的文献；跨境 API 注意数据条款 |
