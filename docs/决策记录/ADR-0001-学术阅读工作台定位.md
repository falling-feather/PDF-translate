# ADR-0001：学术阅读工作台定位

> **状态**：已接受
> **日期**：2026-06-29

## 背景

项目当前已经具备轻量本地 Web 工作台、CLI、分块翻译、术语记忆、串联/并联、精品翻译雏形和资料包输出。导师提出后续希望充分完善并准备专利，因此需要确定长期技术路线。

外部生态中，已有工具和论文覆盖了大量 PDF 原排版翻译能力。若本项目直接追求完整 PDF 回写，会进入字体、坐标、版面重排、公式和图层处理等重型工程问题，且专利叙事容易与已有技术重叠。

## 备选方案

1. **重型 PDF 原排版翻译器**：最大程度保留原 PDF 布局并回写 PDF。
2. **云端 API 包装器**：上传 PDF 后调用翻译 API，输出译文。
3. **学术阅读工作台**：重点做本地部署、结构感知分块、术语记忆、对照阅读、QA 和局部修复。

## 决策

选择方案 3：把项目定位为“轻量、可私有部署、面向学术阅读的 PDF 翻译工作台”。

短期不优先追完整 PDF 原排版回写，而是优先保证：

- 学术结构不被破坏。
- 术语一致。
- 上下文连续。
- 表格和公式有保护。
- 翻译结果可对照、可审查、可局部修复。
- 每次实验可记录指标，为专利交底提供证据。

## 影响

- 后续核心技术路线是 DocumentIR、结构感知分块、动态记忆、结构不变量 QA 和局部修复。
- OCR 的价值是服务于本地结构解析，而不是孤立识别图片。
- 双语 HTML/Markdown 优先级高于 PDF 回写。
- 复杂表格、脚注、图注、人名机构和扫描页需要进入结构层，而不是继续作为普通文本处理。
- PDF 译文可以作为中期输出，但优先从结构化译文重新排版生成，不先追求完全原坐标回填。
- 后端需要从当前大管线函数逐步拆成抽取、分块、翻译、记忆、QA、导出和任务状态模块。

## 复审条件

出现以下情况时重新评估：

- 项目目标改为直接替代 PDFMathTranslate/BabelDOC。
- 已完成结构 IR、QA 和双语输出后，用户强需求集中转向 PDF 回写。
- 导师或专利代理师认为“结构感知学术阅读工作台”路线不具备足够交底价值。
