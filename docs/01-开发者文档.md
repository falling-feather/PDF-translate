## 项目概览

本项目是面向英文论文 PDF 的本地可部署翻译工作台。当前核心闭环是：

```text
上传或指定 PDF
  -> 拆分正文与参考文献
  -> 提取正文页文本与基础富元数据
  -> 生成 DocumentIR、结构分段 manifest、OCR/VLM 路由报告、OCR 调度任务与 OCR 回写报告
  -> 以 1-3 页滑窗分块
  -> 可选译前巡视并写入术语草稿
  -> 调用翻译后端
  -> 更新 memory/ 记忆
  -> 合并 Markdown
  -> 生成译后 QA、局部修复计划、局部修复请求、执行结果与验证结果
  -> 汇总专利实验指标
  -> 生成双语 HTML 阅读版与结构化 PDF 译文
  -> 导出链接索引与 zip 资料包
```

当前项目优势是轻量、可恢复、术语与摘要记忆明确、支持串联/并联/精品翻译。当前短板是 PDF 结构层仍处在原型阶段，复杂表格、公式、多栏阅读顺序、跨页图注/脚注和扫描件 OCR 还没有形成稳定闭环。

## 当前技术栈

- 后端：Python 3.10+、FastAPI、Typer、SQLite、bcrypt、JWT。
- PDF：PyMuPDF，当前以 `page.get_text("text")` 为主。
- 翻译：OpenAI 兼容接口、DeepSeek、Ollama、DeepL、hybrid、echo。
- 前端：Vue 3、Vite，构建后由 FastAPI 静态托管。
- 任务数据：默认写入 `data/app.db` 和 `data/web_jobs/<job_id>/`。

## 核心模块规则

### 翻译管线

**适用范围**：`pipeline.py`、未来 `extractors/`、`chunkers/`、`qa/`、`exporters/`。

**规则**：
1. 新增大功能时优先放到独立模块，避免继续扩大 `run_translate()`。
2. 管线中的每个阶段都要能产出可复查的中间文件或日志。
3. 翻译块输出必须保留 `chunk_id`、页码、后端、prompt 版本等元数据。
4. 串联模式要保护前文摘要、上一块译文尾部和段尾顺延协议。
5. 并联模式不得依赖串联顺延文件，避免跨线程污染上下文。

**原因**：后续要支持结构 IR、QA、局部重译和专利实验数据，必须让管线阶段可拆、可测、可追踪。

**验收**：
- 单个阶段失败时能定位到阶段、块和错误类型。
- 断点续译不会重复已完成块。
- `output/run_log.jsonl`、`output/run_metrics.json`、`state.json`、`chunks_manifest.json` 能解释一次运行。

### 后端注册

**适用范围**：`translators/registry.py`、`translators/factory.py`、`server/routes_web.py`、`settings_service.py`、前端管理端。

**规则**：
1. CLI、Web、管理员后台和用户自定义 API 应读取同一份后端能力表。
2. 能力表至少包含后端 id、展示名称、配置项、是否支持段尾顺延、是否支持并联、是否允许用户临时 key。
3. 不同后端的错误信息应映射为结构化错误码。

**当前落地**：
- `pdf_translate.translators.registry` 统一声明 `echo`、`deepseek`、`openai`、`ollama`、`deepl`、`hybrid` 的 id、展示名称、别名、配置字段、默认启用状态、自定义 API 支持和段尾顺延能力。
- `build_translator()`、CLI help、成本模型名、`_translator_supports_deferral()`、Web 用户后端列表和管理端设置均从 registry 读取后端定义。
- 管理端保存 `enabled_backends` 时会归一化别名并拒绝未知后端；用户端自定义 API 的可选后端由服务端返回，避免前后端口径不一致。

**验收**：
- 新增一个后端时不需要分别改 4-5 个位置。
- 前端展示、后端校验和实际构造器行为一致。

### Web 任务状态

**适用范围**：`server/jobs.py`、`server/routes_web.py`、`server/database.py`、Web 用户端和管理端任务列表。

**规则**：
1. Web 任务运行态以 `JobRegistry` 和 `data/web_jobs/<job_id>/web_status.json` 为主源。
2. `web_status.json` 只记录任务状态、阶段、进度、错误、运行时间、页面统计、任务所有者和原始文件名等运行态快照。
3. SQLite `jobs_meta` 只负责用户归属、任务列表、收藏列表和列表时间，不应承载运行进度。
4. 管线 `output/state.json` 只负责 chunk 断点续跑，不等同于 Web 任务状态。
5. 服务启动恢复任务时，应以 `web_status.json` 所在目录作为 `work_dir`，不要信任快照内可被篡改的路径字段。
6. 用户任务列表和管理员任务列表应通过 `JobRegistry.merge_status_into_rows()` 合并运行态字段，同时保留 SQLite 的 `created_at` 列表时间。
7. 任务列表和任务详情应复用同一份产物摘要，至少暴露原 PDF、译文 Markdown、双语 HTML、结构化 PDF 译文、zip 打包可用性和一致性 warning。
8. 历史任务清理必须先巡检再执行；管理员清理只能删除数据库缺工作目录的行，以及没有数据库索引的任务目录，运行中任务必须跳过。
9. CLI、用户详情和管理员列表应复用同一套诊断字段；`output/state.json` 只说明 chunk 断点，必须以 `pipeline_state_status`、`pipeline_resume_ready` 和 `job_recovery_status` 的形式进入恢复审计。

**当前落地**：
- `JobRecord.to_status_dict()` / `JobRecord.from_status_dict()` 统一任务状态快照结构，并写入 `schema_version: web-job-status-v1`。
- `_persist()` 通过临时文件加 `os.replace()` 原子更新 `web_status.json`，避免写入半截 JSON。
- `/api/user/jobs` 和 `/api/admin/jobs` 已在返回列表时合并运行态字段；历史缺失状态快照的行会返回 `status_available: false`。
- `artifact_fields_for_record()` 会计算 `input_pdf_ready`、`partial_output_ready`、`bilingual_html_ready`、`translated_pdf_ready`、`bundle_zip_ready`、`artifact_consistent`、`artifact_consistency_status` 和 `artifact_warnings`；管理端任务表会展示产物状态，并禁用不可下载的产物按钮。
- `storage_drift()` 会对比 SQLite 索引和 `web_jobs/` 目录，输出数据库缺目录、目录缺数据库索引和运行中任务；`/api/admin/jobs/reconcile` 支持 `apply=false` 巡检和 `apply=true` 清理，并在执行清理时写入审计日志。
- `pipeline_state_fields_for_record()` 会读取 `output/state.json`、`output/chunks_manifest.json` 和 `output/chunks/`，输出已完成 chunk 数、待处理 chunk、续跑可用性、恢复状态和诊断 warning。
- `pdf-translate web-status` 会读取同一份 `web_status.json` 与管线 state 诊断，供 CLI/脚本侧查看 Web 任务恢复状态；`/api/admin/jobs/hydration-report` 会暴露本次启动恢复报告与存储漂移摘要。

**验收**：
- `tests/test_job_status.py` 覆盖状态快照 schema、重启恢复、恢复报告、路径边界、列表合并、缺失状态快照、完成态缺失译文的产物异常、管线 state 诊断、CLI `web-status`、存储漂移识别和路径穿越清理保护。

### 结构化错误码

**适用范围**：`pdf_translate/error_codes.py`、`translators/`、`server/jobs.py`、`run_metrics.py`、`qa/metrics.py`、`frontend/src/views/UserView.vue`。

**规则**：
1. 底层异常应先映射为 `ErrorInfo`，再进入任务状态、日志和指标；不要只向上抛一段不可分类的字符串。
2. `ErrorInfo` 字段包含 `code`、`category`、`retryable`、`user_message`、`next_step`、`source`、`http_status` 和 `exception_type`。
3. 当前稳定错误码包括 `CONFIG_MISSING_API_KEY`、`CONFIG_INVALID_BACKEND`、`CONFIG_INVALID_OPTION`、`HTTP_RATE_LIMIT`、`HTTP_TIMEOUT`、`HTTP_AUTH_ERROR`、`HTTP_CLIENT_ERROR`、`HTTP_SERVER_ERROR`、`HTTP_NETWORK_ERROR`、`API_RESPONSE_INVALID`、`PDF_PARSE_ERROR`、`TASK_CANCELLED`、`PIPELINE_ERROR` 和 `UNKNOWN_ERROR`。
4. Web 任务状态保留旧字段 `error`，同时新增 `error_code`、`error_category`、`error_retryable`、`error_next_step`、`error_source` 和 `error_http_status`，供前端显示下一步建议。
5. 前端应优先显示 `user_message` / `detail` / `message`，同时展示 `next_step` / `error_next_step` 与 `code` / `error_code`，不要把结构化 JSON 原样弹给用户。
6. `run_metrics.json.summary` 和 `experiment_metrics.json.breakdowns` 必须保留错误码/错误类别分布，便于批量实验统计失败原因。

**当前落地**：
- `pdf_translate.error_codes.PdfTranslateError` 是结构化异常入口，`error_info_from_exception()` 负责把 HTTP 状态、超时、网络错误、PDF 解析错误、取消和配置错误映射为稳定错误码。
- `call_with_http_retry()` 的捕获事件会写入 `error_code`、`error_category` 和 `error_retryable`；重试耗尽或非重试 HTTP 错误会抛出 `PdfTranslateError`。
- `build_translator()` 对缺失 API Key 和未知后端抛出结构化配置错误；`server/jobs.py` 会把失败或取消写入 `web_status.json` 并通过 `/api/jobs/{job_id}` 暴露。
- `RunMetricsRecorder.stage()` 会在阶段异常中记录错误码，`build_run_metrics()` 会汇总 `error_code_counts` 和 `error_category_counts`，并由 `experiment_metrics.json` 回链。
- `frontend/src/views/UserView.vue` 会格式化 FastAPI `detail` 对象和扁平错误字段，提交、取消、下载失败提示以及任务卡片可展示用户消息、下一步建议、错误码和 HTTP 状态。

### 记忆与术语

**适用范围**：`memory_store.py`、译前巡视、精品翻译、未来术语确认台。

**规则**：
1. 术语应区分候选、已确认、已否决、冲突待审；当前新增术语默认写为 `candidate`。
2. 术语应保留来源、页码、章节、结构类型和更新时间。
3. 当前块只注入相关术语，避免全量术语挤占上下文窗口。
4. 模型草拟术语不能默认等同于人工确认术语。

**原因**：术语一致性是学术翻译质量的主要杠杆，也是专利方向中的动态记忆反馈证据。

**验收**：
- 同一英文多译、同一中文多源能进入 `pending_review.json` 或等价待审结构。
- 术语导入导出不破坏现有 `glossary.json`。

**当前落地**：
- `merge_glossary_terms_from_survey()` 新增术语时会写入 `status: candidate`。
- 同一英文术语出现不同中文译名时，不静默覆盖原术语，而是写入 `pending_review.json` 的 `glossary_conflict`。
- 多个英文术语共享同一中文译名时，会写入 `shared_translation_review`，供人工确认是否为同义术语。
- 译后 QA 会读取 `pending_review.json`，在相关块中报告 `glossary_translation_conflict`，修复计划会映射为 `review_glossary_conflict`。

### PDF 结构层

**适用范围**：未来 `extractors/`、`DocumentIR`、结构感知分块、QA。

**规则**：
1. 不直接把复杂 PDF 线性化为不可区分的普通文本。
2. 表格、图注、脚注、注释、人名、机构名、公式、标题、参考文献、页眉页脚应有类型边界。
3. 表格必须保留行列关系、表头、单位、脚注和显著性符号，不能被当作普通段落拼接。
4. 图注、脚注、页下注释和人名机构信息应能回到所属图表、段落或页面。
5. 分块不得切断表格、图注、公式环境和跨页未完段落。
6. 数字、公式编号、引用编号、单位和表格维度应作为结构不变量进入 QA。
7. 结构分段必须记录预算来源、切分原因、预算压力、估算 token 和结构关系保护，不能只产出不可解释的 chunk。

**原因**：当前最大质量风险不是“模型不会翻译”，而是模型看到的输入结构已经被破坏。

**验收**：
- 对表格密集论文，输出能保留行列关系或至少给出风险标记。
- QA 报告能指出数字、引用、公式和表格维度异常。

**当前落地**：
- `pdf_translate.extractors.document_ir.extract_document_ir()` 会生成 `output/document_ir.json`。
- `pdf_translate.chunkers.structure.build_structure_chunks()` 会生成 `output/structure_chunks_manifest.json`，记录 `block_ids`、`boundary_fragment_ids`、`structural_relation_ids`、`approx_tokens` 和 `budget` 元数据，并在页面边界高风险或图注/表注/脚注父子关系相邻时保护结构关系不被目标字符数或普通页数阈值提前拆开。
- 表格块会在 `meta.table` 中记录行列、表头、数字 token、confidence 和 warning；结构分段会把表格优先渲染成 Markdown table。
- 图注、表注和脚注会按同页优先、一页跨页回退的规则写入 `parent_id` 与 `meta.parent_relation`；跨页归属会记录 `parent_page_no`、`parent_page_gap`、`cross_page_parent` 和 `cross_page_parent_attempted`，紧跟表格的脚注会标记为 `footnote_for_table`，无法归属的内容会写入 `meta.parent_warning`。
- 文本块会用本地规则提取人名、机构、模型/数据集名和缩写候选，写入 `meta.entities`；这些结果只表示候选实体，不等同于人工确认术语。
- `pdf_translate.qa.structure.write_structure_qa()` 会生成 `output/structure_qa.json`，记录表格维度、续表风险、图注/表注/表格脚注/脚注关系、实体候选、孤立关系统计、跨页关系成功/孤立统计、页级风险、页面边界碎片和结构块统计。
- `pdf_translate.qa.table_reconstruction.write_table_reconstruction_report()` 会生成 `output/table_reconstruction.json`，按源表格输出单元格坐标、表头、行头、数字/单位/显著性 token、表注/脚注表级绑定、`footnote_bindings` 脚注单元格级绑定证据、ragged 行/空单元格统计、`merged_cell_candidates` 合并单元格候选证据和续表关系；跨页续表会额外进入 `continued_table_groups`，记录续表片段链、兼容性检查、合并/拒绝状态、拒绝/警告原因分布、合并后行列视图、重复表头跳过数、组级合并单元格候选和组级可重建状态，作为表格保护式翻译、PDF 输出和专利实验的前置证据。
- `pdf_translate.qa.table_reconstruction.build_table_translation_hints()` 会把当前 chunk 覆盖的表格证据压缩为 `TranslationRequest.structure_hints`，OpenAI/DeepSeek 兼容后端会以 `v5`「结构保护提示」注入模型消息，要求保留 Markdown 表格行列、表头/行头关系、脚注单元格绑定、续表合并关系、疑似合并单元格候选和锁定 token；合并单元格候选必须以“未确认结构风险”处理，不得提示模型自行声称已经完成合并或重排。
- `pdf_translate.qa.table_reconstruction.write_structure_hints_manifest()` 会生成 `output/structure_hints_manifest.json`，把每个 chunk 实际注入翻译请求的 `structure_hints` 固化为审计清单，记录覆盖页码、块 ID、表格 ID、续表组 ID、提示文本、提示长度、空提示 chunk 数、脚注绑定数、锁定 token 数以及合并单元格候选类型/原因分布；串行和并行翻译路径都从该清单复用提示文本，避免“记录的提示”和“实际发送的提示”不一致。
- `pdf_translate.qa.chunk_boundary.write_chunk_boundary_qa()` 会生成 `output/chunk_boundary_qa.json`，对照结构 QA 的 `page_boundary_fragments` 与当前实际 chunks，统计高风险页面边界是被保护、同块覆盖还是被切开，并汇总 `budget_split_reason_counts`、`budget_pressure_counts`、`budget_overflow_chunk_count` 和 `structural_relation_protected_count`。

- 与公式 OCR 结构契约和本地门禁相关的 `ocr_formula_context_task_count`、`ocr_formula_context_ready_task_count`、`ocr_formula_context_writeback_count`、`ocr_formula_context_candidate_count`、`ocr_formula_latex_candidate_count`、`ocr_formula_tokens_candidate_count`、`ocr_equation_labels_candidate_count`、`ocr_formula_confidence_candidate_count`、`ocr_result_formula_latex_count`、`ocr_result_formula_token_count`、`ocr_result_equation_label_count`、`ocr_structured_formula_candidate_count`、`ocr_structured_formula_gate_passed_count`、`ocr_structured_formula_gate_review_count`、`ocr_structured_formula_gate_blocked_count`、`ocr_structured_formula_missing_locked_token_count`、`ocr_formula_context_task_rate`、`ocr_formula_context_ready_rate`、`ocr_structured_formula_gate_pass_rate`、`ocr_structured_formula_gate_review_rate` 和 `ocr_candidate_structured_formula_gate_issue_counts` 会进入 `experiment_metrics.json`，用于观察公式 OCR 是否保留 LaTeX、公式 token、编号和置信度，并让低置信或缺失锁定 token 的候选进入复核。
- `pdf_translate.run_metrics.RunMetricsRecorder` 会维护 `output/run_log.jsonl` 和 `output/run_metrics.json`；前者按事件记录阶段耗时、chunk 翻译、跳过块、错误码和增量合并，后者汇总总耗时、阶段耗时、请求字符数、译文字符数、估算 token、翻译请求次数、HTTP 尝试次数、HTTP 重试次数、错误码分布和吞吐指标，并被 `experiment_metrics.json.performance` / `experiment_metrics.json.breakdowns` 回链。
- `pdf_translate.translators.http_retry.capture_http_retry_events()` 可在翻译块外层捕获 `call_with_http_retry()` 的尝试事件；事件只记录上下文、尝试序号、状态码、错误类型、结构化错误码、耗时和是否继续重试，不记录请求正文、API Key 或模型响应正文。
- `pdf_translate.costing.write_cost_estimate()` 会读取可配置成本画像，生成 `output/cost_estimate.json`；默认只内置 `echo` / `ollama` 的零外部 API 成本，不硬编码 DeepSeek/OpenAI/DeepL 等会变化的价格。命中画像后会按估算请求 token、估算响应 token、字符数和可计费请求次数计算 `estimated_total_cost`；可计费请求数优先使用 HTTP 尝试次数，缺失时退回成功翻译请求次数，并被 `experiment_metrics.json.performance` 回链。
- `PageIR.meta` 会记录文本字符数、文本区域比例、图像区域比例和块类型计数，供 OCR/VLM 路由、QA 和后续输出层复用。
- `pdf_translate.vision.routing.write_vision_route()` 会生成 `output/vision_route.json`，按页面给出 `text_only`、`local_ocr`、`vlm_review` 或 `skip_blank` 建议和触发原因；对非 `text_only` 页面会尽量渲染 `output/vision_pages/page-0001.png` 这类页面预览，并按 DocumentIR 块裁剪 `output/vision_crops/page-0001/p1-b0000-image.png` 这类候选区域，作为后续本地 OCR、VLM 调度和导师评审的可视证据。
- `pdf_translate.vision.ocr_tasks.write_ocr_task_manifest()` 会生成 `output/ocr_tasks.json`，把 OCR/VLM 路由页和候选区域裁剪证据转成待执行任务，记录输入图、优先级、建议本地 OCR 引擎、VLM fallback、状态和 DocumentIR 回写目标。
- 表格类 OCR 任务会额外写入 `layout_scope=table_region`、`target_structure_type=table`、`table_context`、`structure_contract` 和 `writeback.subtarget`。其中 `table_context` 记录源表格 ID、页码、bbox、行列数、表头、数字 token、locked token、源置信度和 warning；`structure_contract` 声明表格 OCR 可选返回 `structured_cells`、`cell_bboxes`、`merged_cell_candidates` 与 `table_footnotes` 等结构字段。该契约保持向后兼容，不改变 `ocr-result-v1` 的必填字段。
- 公式类 OCR 任务会额外写入 `layout_scope=formula_region`、`target_structure_type=formula`、`formula_context`、`structure_contract` 和 `writeback.subtarget`。其中 `formula_context` 记录源公式 ID、页码、bbox、源文本、抽取到的公式 token 和 locked token；`structure_contract` 声明公式 OCR 可选返回 `formula_latex`、`formula_tokens`、`equation_labels` 与 `formula_confidence`，供公式 OCR、手工标注或后续公式解析器接入同一套回写门禁。
- `pdf_translate.vision.ocr_executor.execute_ocr_tasks()` 可在 CLI 显式开启 `--execute-ocr` 且未传入 `--ocr-results` 时消费 `ocr_tasks.json`，默认尝试调用本地 `tesseract` CLI；执行成功、失败、跳过、缺少二进制或缺少输入图都会进入 `ocr-results-v1` 的 `execution` 摘要，不直接写回 DocumentIR。
- `pdf_translate.vision.ocr_writeback.write_ocr_results_payload()` 会生成 `output/ocr_results.json`，把外部或本地 OCR 执行器传入的 `ocr-results-v1` JSON 规范化为结果载荷，保留来源路径、结果数、无效结果数、状态分布和引擎分布；未提供结果时会写出空载荷而不是跳过证据链。
- `pdf_translate.vision.ocr_writeback.write_ocr_writeback()` 会生成 `output/ocr_writeback.json` 和 `output/document_ir_ocr.json`，消费 `output/ocr_results.json` 后把通过置信度与状态门禁的文本追加到 page/block 的 `meta.ocr_candidates`；默认没有真实 OCR 结果时会输出 pending 报告和未污染的增强 IR 副本。该阶段不替换 `DocumentIR.text`、不改写翻译 chunks。
- `pdf_translate.qa.ocr_candidates.write_ocr_candidate_qa()` 会生成 `output/ocr_candidate_qa.json` 和 `output/ocr_candidate_qa.md`，把 OCR 候选按 `candidate`、`needs_review`、`blocked` 分类，阻断过短、低有效字符比例、低置信度、重复源文本和需要表格/公式结构复核的候选。
- `pdf_translate.vision.ocr_promotion.write_ocr_candidate_promotion()` 会生成 `output/ocr_candidate_promotion.json`、`output/ocr_candidate_promotion.md` 和 `output/document_ir_promoted.json`；只有 OCR 候选 QA 中 `status=candidate` 的文本会进入晋级 IR。结构分段使用晋级 IR，原始 `document_ir.json` 与增强 `document_ir_ocr.json` 保持为证据基线，不被覆盖。
- OCR 回写、候选 QA 和候选晋级会透传 `target_structure_type`、`table_context`、`formula_context`、`subtarget`、`structure_contract` 以及 OCR 结果中的 `structured_cells`、`cell_bboxes`、`merged_cell_candidates`、`table_footnotes`、`formula_latex`、`formula_tokens`、`equation_labels` 和 `formula_confidence`。因此即使当前仍是 append-only 候选写回，也能保留后续单元格级晋级、公式级晋级、表格/公式修复和高精度 OCR 引擎接入所需的结构痕迹。旧结果文件不含这些字段时仍按普通 OCR 文本候选处理。
- 对 `target_structure_type=table` 的结构化 OCR 候选，`ocr_candidate_qa` 会运行本地表格结构门禁：检查 `structured_cells` 是否为非空列表、单元格行列坐标是否可解析、行列范围是否与 `table_context.row_count` / `column_count` 一致、`cell_bboxes` 覆盖是否足够、`table_context.locked_tokens` 是否能在单元格文本中找到。门禁通过的表格 OCR 候选不再自动追加 `needs_table_structure_review`；缺结构、缺 bbox、缺 locked token、行列不一致或引擎 warning 仍会进入复核或阻断。
- 对 `target_structure_type=formula` 的结构化 OCR 候选，`ocr_candidate_qa` 会运行本地公式结构门禁：检查候选是否返回可读文本、`formula_latex`、`formula_tokens`、`equation_labels` 与 `formula_confidence`，并把 `formula_context.source_tokens` / `locked_tokens` 中缺失的符号、变量或公式编号列入复核原因。门禁通过的公式 OCR 候选可作为后续公式保护翻译和公式邻近段落修复证据；缺 LaTeX、缺 token、缺编号、低置信或缺 locked token 的候选不会静默晋级。
- CLI 可用 `--chunk-strategy structure` 试用结构分段翻译路径；默认仍为 `page`，避免直接改变 Web 用户路径。CLI 也可通过 `--ocr-results <json>` 接收外部 OCR 结果文件，或通过 `--execute-ocr --ocr-command <cmd>` 尝试执行本地 OCR 任务，供本地 OCR 原型或人工实验回灌使用。

**QA 边界**：
- 当前译后 QA 是纯规则检查，不调用模型，不直接修改译文。
- QA JSON 面向实验统计，QA Markdown 面向导师评审和人工复核。
- 结构边界检测会在翻译前检测相邻页面首尾块的断句、续表和接续风险，QA 输出 `page_boundary_fragments` 与 `table_continuations`，结构分段清单输出 `boundary_fragment_ids`、`structural_relation_ids` 和 `budget`，分段边界 QA 输出 `split_boundary_count`、`protected_boundary_count`、`high_risk_split_count`、`budget_split_reason_counts`、`budget_pressure_counts`、`budget_overflow_chunk_count` 和 `structural_relation_protected_count`，策略对比输出 `active_split_reduction_vs_baseline`。
- 关系归属检测会在 `relationships` 中记录 caption/footnote 的父块、关系类型、父页码、页距、跨页标记和孤立 warning，用于后续图文重排、PDF 输出和结构保持率统计；跨页回退当前只允许相邻 1 页，避免误把远距离图表强行绑定。
- 表格重建报告会在翻译前保留表格单元格上下文、表级脚注、脚注单元格级绑定证据和合并单元格候选证据，跨页续表会先经兼容性闸门再生成组级合并视图，并把拒绝原因和警告原因同时聚合为原始分布与稳定类别分布；译后 QA 会把锁定 token 缺失定位到表格单元格坐标，但合并单元格当前仍是候选证据层，高精度识别、重排和单元格级重译执行仍需后续实现。
- 实体候选检测会在 `entities` 中记录块 ID、页码、实体类型、文本、置信度和规则来源，用于后续术语确认台、实体一致性 QA 和专利实验统计。
- 译后实体 QA 会检查候选实体是否在译文中保留；当前只报告缺失候选，不判断中文译名是否正确。
- 译后表格 QA 会优先用 Markdown 表格形状比对；当源 chunk 不是 Markdown 表格但 DocumentIR 中存在表格块时，会按页码或结构 chunk 的 `block_ids` 读取源表格行列数，发现译文没有保留 Markdown 表格或行列不一致时输出 `table_shape_mismatch`；当目标表格对应单元格缺失数字、单位或显著性符号等锁定 token 时输出 `table_cell_token_mismatch`。
- 修复计划会读取 QA JSON 中的异常块，给出优先级、修复范围、执行器和证据；修复请求会继续读取源 chunk 与当前译文，生成可由后端执行的局部修复 payload，并为表格/单元格问题记录目标表格索引、表格 ID 与目标单元格摘要；修复结果只记录执行状态与候选修复片段；修复验证只判断候选片段是否满足本地不变量；修复合并只写独立副本和合并报告，不覆盖正式译文。
- 实验指标不会替代明细文件；它只聚合可复现实验所需的摘要、比例和证据文件路径，后续批量样本对比应读取该文件并回链到结构 QA、结构提示清单、分段边界 QA、译后 QA、修复计划、修复请求、修复结果、修复验证、修复合并和合并后 QA 等明细 JSON。
- 成本估算只在命中成本画像时给出金额；未配置画像时仍保留 token、字符、翻译请求次数、HTTP 尝试次数和可计费请求次数，不得把未配置价格的 `0` 误读为真实免费。失败 HTTP 尝试的 token/字符是否计费取决于服务商账单规则，当前只把 `per_request` 按尝试次数计入，失败尝试 token 成本仍需用真实账单校准。
- 术语一致性 QA 目前可检查源文出现的英文术语是否使用术语库指定中文译名，并能暴露待审术语冲突；候选确认流程、置信度和跨章节作用域仍待完善。
- 当前修复执行不会自动合并回正式译文；后续自动修复器应消费 `repair_results.json`、`repair_validation.json`、`repair_merge.json`、`repair_merge_qa.json` 与 `output/repairs/`，在人工确认或策略通过后再替换正式译文，并记录失败回退。

**阅读输出当前落地**：
- `pdf_translate.exporters.bilingual_html.write_bilingual_html()` 会生成 `output/bilingual.html`。
- 双语 HTML 按 chunk 展示原文与译文，并在同页展示 QA 问题和局部修复建议。
- `pdf_translate.exporters.translated_pdf.write_translated_pdf()` 会生成 `output/translated_full.pdf` 和 `output/translated_pdf_report.json`，按 chunk 重排标题、段落、Markdown 表格、QA 问题和修复建议，并把页数、chunk 数、表格数、QA 数、caption/footnote 跨页关系、表格脚注单元格绑定、续表组和结构关系输出证据写入报告。
- 当前 HTML 和 PDF 都是评审/对照阅读产物，不追求 PDF 原坐标回填；后续 PDF 输出应继续从结构化块和修复后译文重新排版，补齐复杂图表、脚注和页码关系。

### 图像与 OCR

**适用范围**：未来 `extractors/`、OCR 适配器、页面截图、VLM 路由、图文 QA。

**规则**：
1. 不能只依赖“本页文本数量低于平均值”判断是否需要图像模型；该规则只能作为低成本启发。
2. 页面应保留文本层、图片区域、页面截图、图注和 OCR 置信度等信息。
3. 扫描页、图片型表格、公式密集页和图注密集页应进入风险列表。
4. 本地 OCR 优先服务于结构识别和图文边界恢复，不把图片翻译作为第一目标。

**原因**：图像能力的价值在于避免漏掉扫描页、图表和图注，而不是简单提高模型调用次数。

**验收**：
- 低文本页能区分扫描页、图片页、公式页和普通空白页。
- 开启 OCR/VLM 后，产物能说明哪些页面调用了图像处理，原因是什么，并能回链到对应页面预览。

**当前落地**：
- 管线会生成 `output/vision_route.json`，基于文本字符数、文本区域比例、图像数量、图像区域比例、图注/表格/公式块和页级 warning 给出路由建议。
- 非 `text_only` 页面会生成 `output/vision_pages/` 下的页面预览图，并在 `vision_route.json.pages[].evidence` 中记录路径、渲染状态、尺寸和缩放比例。
- 非 `text_only` 页面还会根据 `image`、`table`、`caption`、`formula` 等 DocumentIR 块生成 `output/vision_crops/` 区域裁剪图，并在 `region_crops` 中记录块 ID、块类型、bbox、裁剪路径和尺寸。
- `output/ocr_tasks.json` 会把区域裁剪或页面预览进一步转成 `pending_engine` 任务；没有可视证据的任务会标记为 `blocked_missing_visual_evidence`，避免静默丢失 OCR 候选。
- 当前已完成本地路由、页面预览、候选区域裁剪、OCR 调度任务、tesseract CLI 本地 OCR 执行入口、OCR 结果文件入口、OCR 结果回写报告、OCR 候选 QA、OCR 候选晋级报告和晋级 DocumentIR 副本；仍未接入 VLM。后续 OCR 适配器应输出 `ocr-results-v1` payload，由 `output/ocr_results.json` 固化后交给回写门禁追加为候选文本，再经 QA 和晋级门禁进入结构分段。

### 输出形态

**适用范围**：`exporters/`、Markdown 合并、双语 HTML 和 PDF 导出。

**规则**：
1. Markdown 继续作为轻量产物，但复杂论文不能只依赖纯 Markdown。
2. 双语 HTML 应优先保留结构、页码、块边界、图表占位和术语 hover。
3. PDF 输出应由结构化译文重新排版生成，不宜一开始追求完全回填原 PDF 坐标。
4. 表格、注释、人名、图注和引用在 HTML/PDF 中必须有清晰视觉层级。

**原因**：当前纯 Markdown 在表格、注释和图文混排场景下可读性不足，输出层必须承接结构识别的成果。

**验收**：
- 同一任务至少能输出结构化 Markdown、阅读版 HTML 和结构化 PDF 译文。
- PDF 输出原型能保留标题、段落、表格、QA 标记和基础页码关系；复杂图表、脚注和原版式关系进入后续验收。

## 推荐演进模块

后续新增模块建议按以下边界落地：

```text
pdf_translate/
  extractors/       # PDF/OCR/版面结构抽取，输出 DocumentIR
  chunkers/         # 页级、段落级、章节级、结构感知分块
  translators/      # 翻译服务适配与能力注册
  memory/           # 术语库、翻译记忆、摘要、冲突处理
  qa/               # 数字/公式/引用/术语/表格不变量检查
  exporters/        # Markdown、双语 HTML、结构化 PDF、zip 输出
  jobs/             # 队列、状态、取消、恢复、限流
  server/           # FastAPI 路由层
```

迁移不需要一次完成。新能力优先放进新目录，旧代码在有测试保护后逐步搬迁。

## 安全边界

- 生产环境必须设置管理员引导密码，不应依赖默认值。
- 对公网服务应显式配置 `PDF_TRANSLATE_CORS_ORIGINS`。
- `pdf_translate.server.security_preflight` 负责生产安全预检，CLI `pdf-translate security-check` 与管理端 `/api/admin/security/preflight` 必须复用同一份报告口径。
- JWT 必须带 `exp` 过期声明，登录有效期由 `PDF_TRANSLATE_JWT_TTL_MINUTES` 控制；服务端解码时必须拒绝过期 token 和缺少 `exp` 的历史 token。
- 当 `PDF_TRANSLATE_ENV` / `PDF_TRANSLATE_DEPLOYMENT_MODE` 进入生产模式时，Web app 初始化必须调用生产安全门禁，存在高/中风险配置时直接阻断启动。
- 上传大小由 `PDF_TRANSLATE_MAX_UPLOAD_MB` 控制，默认 120MB；Web 上传接口与安全预检必须使用同一套解析逻辑。上传保存必须流式读取，先校验 `%PDF-` 魔数，再按块写入任务目录，超限或非法文件不得留下半截输入文件。
- 管理端保存的 API Key 落在本地 SQLite；设置快照只返回 `secret_fields` 配置状态，不得回传密钥明文。设置 `PDF_TRANSLATE_SECRET_KEY` 或 `PDF_TRANSLATE_SECRET_KEY_FILE` 后，新保存的 API Key 必须以 `enc:v1:` 前缀密文落盘；旧明文值保持只读兼容，并由安全预检提示管理员重新保存加密。
- 用户临时 API Key 应只用于本次任务，不应默认落盘。
- 上传文件、输出文件和 zip 打包路径必须始终限制在任务工作目录内。zip 资料包遍历必须跳过 symlink，并确认候选文件 `resolve()` 后仍位于任务目录下。
