> **文档定位**：说明当前系统架构、模块边界、扩展方式和长期开发规则。
> **适用范围**：`pdf_translate/` 后端、`frontend/` 前端、Web 任务工作流与 CLI 工作流。
> **最后更新**：2026-06-29
> **维护者**：项目负责人
> **关联文档**：[更新规划](02-更新规划.md)、[测试与验收指南](05-测试与验收指南.md)、[安装与 API 配置](../SETUP_MANUAL.md)

## 项目概览

本项目是面向英文论文 PDF 的本地可部署翻译工作台。当前核心闭环是：

```text
上传或指定 PDF
  -> 拆分正文与参考文献
  -> 提取正文页文本与基础富元数据
  -> 生成 DocumentIR、结构分段 manifest 与 OCR/VLM 路由报告
  -> 以 1-3 页滑窗分块
  -> 可选译前巡视并写入术语草稿
  -> 调用翻译后端
  -> 更新 memory/ 记忆
  -> 合并 Markdown
  -> 生成译后 QA 与局部修复计划
  -> 生成双语 HTML 阅读版
  -> 导出链接索引与 zip 资料包
```

当前项目优势是轻量、可恢复、术语与摘要记忆明确、支持串联/并联/精品翻译。当前短板是 PDF 仍主要被处理为文本流，表格、公式、图注、多栏阅读顺序和扫描件 OCR 还没有形成统一结构层。

## 当前技术栈

- 后端：Python 3.10+、FastAPI、Typer、SQLite、bcrypt、JWT。
- PDF：PyMuPDF，当前以 `page.get_text("text")` 为主。
- 翻译：OpenAI 兼容接口、DeepSeek、Ollama、DeepL、hybrid、echo。
- 前端：Vue 3、Vite，构建后由 FastAPI 静态托管。
- 任务数据：默认写入 `data/app.db` 和 `data/web_jobs/<job_id>/`。

## 目录边界

| 路径 | 职责 | 注意事项 |
| --- | --- | --- |
| `pdf_translate/pipeline.py` | 当前主翻译管线 | 职责偏多，后续新增能力优先拆到新模块 |
| `pdf_translate/chunking.py` | 固定页数滑窗分块 | 后续结构感知分块不要直接塞进该函数 |
| `pdf_translate/extractors/` | 结构化 PDF 抽取，当前输出轻量 `DocumentIR` | 已有基础块类型识别，后续扩展 OCR/版面解析 |
| `pdf_translate/chunkers/` | 基于结构数据构建翻译块 | 当前有实验性结构分段器 |
| `pdf_translate/vision/` | 基于 DocumentIR 生成页面级 OCR/VLM 路由建议 | 当前只输出本地风险报告，不直接调用 OCR/VLM |
| `pdf_translate/pdf_structure.py` | 正文/参考文献拆分 | 当前基于标题启发式和尾部兜底 |
| `pdf_translate/memory_store.py` | 术语、摘要、风格、待确认项、段尾顺延 | 后续需要加入术语状态、置信度和作用范围 |
| `pdf_translate/survey.py` | 译前巡视、术语草拟、风险标签 | 目前只基于文本和图片/链接计数保守估计 |
| `pdf_translate/exporters/` | 面向阅读和评审的本地输出生成 | 当前输出双语 HTML，未来扩展 PDF 译文 |
| `pdf_translate/translators/` | 翻译后端适配 | 建议引入统一 backend registry |
| `pdf_translate/server/` | FastAPI、用户、任务、管理端、审计 | 任务状态目前分布在内存、JSON、SQLite 和管线 state |
| `frontend/` | Web 工作台源码 | 修改后需重新构建到 `pdf_translate/server/static/` |
| `docs/` | 正式项目文档 | `plans/` 只作为本地草稿，不作为协作入口 |

## 核心模块规则

### 翻译管线

**适用范围**：`pipeline.py`、未来 `extractors/`、`chunkers/`、`qa/`、`exporters/`。

**规则**：
1. 新增大功能时优先放到独立模块，避免继续扩大 `run_translate()`。
2. 管线中的每个阶段都要能产出可复查的中间文件或日志。
3. 翻译块输出必须保留 `chunk_id`、页码、后端、prompt 版本等元数据。
4. 串联模式要保护前文摘要、上一块译文尾部和段尾顺延协议。
5. 并联模式不得依赖串联顺延文件，避免跨线程污染上下文。

**原因**：后续要支持结构 IR、QA、局部重译和专利实验数据，必须让管线阶段可拆、可测、可追踪。

**验收**：
- 单个阶段失败时能定位到阶段、块和错误类型。
- 断点续译不会重复已完成块。
- `output/run_log.jsonl`、`state.json`、`chunks_manifest.json` 能解释一次运行。

### 后端注册

**适用范围**：`translators/factory.py`、`server/routes_web.py`、`settings_service.py`、前端管理端。

**规则**：
1. CLI、Web、管理员后台和用户自定义 API 应读取同一份后端能力表。
2. 能力表至少包含后端 id、展示名称、配置项、是否支持段尾顺延、是否支持并联、是否允许用户临时 key。
3. 不同后端的错误信息应映射为结构化错误码。

**原因**：当前翻译工厂支持多后端，但 Web 可选后端主要暴露 `echo/deepseek`，配置边界不够统一。

**验收**：
- 新增一个后端时不需要分别改 4-5 个位置。
- 前端展示、后端校验和实际构造器行为一致。

### 记忆与术语

**适用范围**：`memory_store.py`、译前巡视、精品翻译、未来术语确认台。

**规则**：
1. 术语应区分候选、已确认、已否决、冲突待审。
2. 术语应保留来源、页码、章节、结构类型和更新时间。
3. 当前块只注入相关术语，避免全量术语挤占上下文窗口。
4. 模型草拟术语不能默认等同于人工确认术语。

**原因**：术语一致性是学术翻译质量的主要杠杆，也是专利方向中的动态记忆反馈证据。

**验收**：
- 同一英文多译、同一中文多源能进入 `pending_review.json` 或等价待审结构。
- 术语导入导出不破坏现有 `glossary.json`。

### PDF 结构层

**适用范围**：未来 `extractors/`、`DocumentIR`、结构感知分块、QA。

**规则**：
1. 不直接把复杂 PDF 线性化为不可区分的普通文本。
2. 表格、图注、脚注、注释、人名、机构名、公式、标题、参考文献、页眉页脚应有类型边界。
3. 表格必须保留行列关系、表头、单位、脚注和显著性符号，不能被当作普通段落拼接。
4. 图注、脚注、页下注释和人名机构信息应能回到所属图表、段落或页面。
5. 分块不得切断表格、图注、公式环境和跨页未完段落。
6. 数字、公式编号、引用编号、单位和表格维度应作为结构不变量进入 QA。

**原因**：当前最大质量风险不是“模型不会翻译”，而是模型看到的输入结构已经被破坏。

**验收**：
- 对表格密集论文，输出能保留行列关系或至少给出风险标记。
- QA 报告能指出数字、引用、公式和表格维度异常。

**当前落地**：
- `pdf_translate.extractors.document_ir.extract_document_ir()` 会生成 `output/document_ir.json`。
- `pdf_translate.chunkers.structure.build_structure_chunks()` 会生成 `output/structure_chunks_manifest.json`。
- 表格块会在 `meta.table` 中记录行列、表头、数字 token、confidence 和 warning；结构分段会把表格优先渲染成 Markdown table。
- `pdf_translate.qa.structure.write_structure_qa()` 会生成 `output/structure_qa.json`，记录表格维度、页级风险和结构块统计。
- `pdf_translate.qa.translation.write_translation_qa()` 会生成 `output/qa_report.json` 和 `output/qa_report.md`，检查数字、引用、图表编号、表格形状、重复段落和英文残留。
- `pdf_translate.qa.repair.write_repair_plan()` 会生成 `output/repair_plan.json` 和 `output/repair_plan.md`，把 QA 问题映射成块级、段落级或表格级修复建议。
- `PageIR.meta` 会记录文本字符数、文本区域比例、图像区域比例和块类型计数，供 OCR/VLM 路由、QA 和后续输出层复用。
- `pdf_translate.vision.routing.write_vision_route()` 会生成 `output/vision_route.json`，按页面给出 `text_only`、`local_ocr`、`vlm_review` 或 `skip_blank` 建议和触发原因。
- CLI 可用 `--chunk-strategy structure` 试用结构分段翻译路径；默认仍为 `page`，避免直接改变 Web 用户路径。

**QA 边界**：
- 当前译后 QA 是纯规则检查，不调用模型，不直接修改译文。
- QA JSON 面向实验统计，QA Markdown 面向导师评审和人工复核。
- 修复计划会读取 QA JSON 中的异常块，给出优先级、修复范围、执行器和证据。
- 当前修复计划不直接触发模型重译；后续自动修复器应消费 `repair_plan.json`，按块、段落或表格单元格局部重译。

**阅读输出当前落地**：
- `pdf_translate.exporters.bilingual_html.write_bilingual_html()` 会生成 `output/bilingual.html`。
- 双语 HTML 按 chunk 展示原文与译文，并在同页展示 QA 问题和局部修复建议。
- 当前 HTML 是评审/对照阅读产物，不追求 PDF 原坐标回填；后续 PDF 输出应从结构化块和修复后译文重新排版生成。

### 图像与 OCR

**适用范围**：未来 `extractors/`、OCR 适配器、页面截图、VLM 路由、图文 QA。

**规则**：
1. 不能只依赖“本页文本数量低于平均值”判断是否需要图像模型；该规则只能作为低成本启发。
2. 页面应保留文本层、图片区域、页面截图、图注和 OCR 置信度等信息。
3. 扫描页、图片型表格、公式密集页和图注密集页应进入风险列表。
4. 本地 OCR 优先服务于结构识别和图文边界恢复，不把图片翻译作为第一目标。

**原因**：图像能力的价值在于避免漏掉扫描页、图表和图注，而不是简单提高模型调用次数。

**验收**：
- 低文本页能区分扫描页、图片页、公式页和普通空白页。
- 开启 OCR/VLM 后，产物能说明哪些页面调用了图像处理，原因是什么。

**当前落地**：
- 管线会生成 `output/vision_route.json`，基于文本字符数、文本区域比例、图像数量、图像区域比例、图注/表格/公式块和页级 warning 给出路由建议。
- 当前仅完成本地路由与证据报告，不直接调用 OCR 引擎、页面截图或 VLM；后续 OCR 适配器应把识别结果回写到 DocumentIR，而不是生成孤立文本。

### 输出形态

**适用范围**：`exporters/`、Markdown 合并、未来双语 HTML 和 PDF 导出。

**规则**：
1. Markdown 继续作为轻量产物，但复杂论文不能只依赖纯 Markdown。
2. 双语 HTML 应优先保留结构、页码、块边界、图表占位和术语 hover。
3. PDF 输出应由结构化译文重新排版生成，不宜一开始追求完全回填原 PDF 坐标。
4. 表格、注释、人名、图注和引用在 HTML/PDF 中必须有清晰视觉层级。

**原因**：当前纯 Markdown 在表格、注释和图文混排场景下可读性不足，输出层必须承接结构识别的成果。

**验收**：
- 同一任务至少能输出结构化 Markdown 和阅读版 HTML。
- PDF 输出原型能保留标题、段落、表格、图注和页码关系。

## 推荐演进模块

后续新增模块建议按以下边界落地：

```text
pdf_translate/
  extractors/       # PDF/OCR/版面结构抽取，输出 DocumentIR
  chunkers/         # 页级、段落级、章节级、结构感知分块
  translators/      # 翻译服务适配与能力注册
  memory/           # 术语库、翻译记忆、摘要、冲突处理
  qa/               # 数字/公式/引用/术语/表格不变量检查
  exporters/        # Markdown、双语 HTML、zip、未来 PDF 输出
  jobs/             # 队列、状态、取消、恢复、限流
  server/           # FastAPI 路由层
```

迁移不需要一次完成。新能力优先放进新目录，旧代码在有测试保护后逐步搬迁。

## 本地开发

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -e .
pdf-translate --help
```

前端构建：

```bash
cd frontend
npm install
npm run build
```

启动 Web：

```bash
pdf-translate-web
```

更多环境变量和 API 配置见 [SETUP_MANUAL.md](../SETUP_MANUAL.md)。

## 安全边界

- 生产环境必须设置管理员引导密码，不应依赖默认值。
- 对公网服务应显式配置 `PDF_TRANSLATE_CORS_ORIGINS`。
- 管理端保存的 API Key 当前落在本地 SQLite；面向多人服务时需要明确提示或接入加密/密钥管理。
- 用户临时 API Key 应只用于本次任务，不应默认落盘。
- 上传文件、输出文件和 zip 打包路径必须始终限制在任务工作目录内。
