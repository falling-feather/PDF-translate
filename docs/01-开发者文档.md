## 项目概览

本项目是面向英文论文 PDF 的本地可部署翻译工作台。当前核心闭环是：

```text
上传或指定 PDF
  -> 拆分正文与参考文献
  -> 提取正文页文本与基础富元数据
  -> 生成 DocumentIR、结构分段 manifest、OCR/VLM 路由报告、OCR 调度任务与 OCR 回写报告
  -> 以 1-3 页滑窗分块
  -> 可选译前巡视并写入术语草稿
  -> 调用翻译后端
  -> 更新 memory/ 记忆
  -> 合并 Markdown
  -> 生成译后 QA、结构关系保真检查、局部修复计划、局部修复请求、执行结果、验证结果、补丁审核、修复效果对比、发布确认和回滚演练报告
  -> 汇总专利实验指标
  -> 生成双语 HTML 阅读版与结构化 PDF 译文
  -> 导出链接索引与 zip 资料包
```

当前项目优势是轻量、可恢复、术语与摘要记忆明确、支持串联/并联/精品翻译。当前短板是 PDF 结构层仍处在原型阶段，复杂表格、公式、多栏阅读顺序、跨页图注/脚注和扫描件 OCR 还没有形成稳定闭环。

## 当前技术栈

- 后端：Python 3.10+、FastAPI、Typer、SQLite、bcrypt、JWT。
- PDF：PyMuPDF，当前以 `page.get_text("text")` 为主。
- 翻译：OpenAI 兼容接口、DeepSeek、Ollama、DeepL、hybrid、echo。
- 前端：Vue 3、Vite，构建后由 FastAPI 静态托管。
- 任务数据：默认写入 `data/app.db` 和 `data/web_jobs/<job_id>/`。

## 核心模块规则

### 翻译管线

**适用范围**：`pipeline.py`、未来 `extractors/`、`chunkers/`、`qa/`、`exporters/`。

**规则**：
1. 新增大功能时优先放到独立模块，避免继续扩大 `run_translate()`。
2. 管线中的每个阶段都要能产出可复查的中间文件或日志。
3. 翻译块输出必须保留 `chunk_id`、页码、后端、prompt 版本等元数据。
4. 串联模式要保护前文摘要、上一块译文尾部和段尾顺延协议。
5. 并联模式不得依赖串联顺延文件，避免跨线程污染上下文。

**原因**：后续要支持结构 IR、QA、局部重译和专利实验数据，必须让管线阶段可拆、可测、可追踪。

**验收**：
- 单个阶段失败时能定位到阶段、块和错误类型。
- 断点续译不会重复已完成块。
- `output/run_log.jsonl`、`output/run_metrics.json`、`state.json`、`chunks_manifest.json` 能解释一次运行。

### 后端注册

**适用范围**：`translators/registry.py`、`translators/factory.py`、`server/routes_web.py`、`settings_service.py`、前端管理端。

**规则**：
1. CLI、Web、管理员后台和用户自定义 API 应读取同一份后端能力表。
2. 能力表至少包含后端 id、展示名称、配置项、是否支持段尾顺延、是否支持并联、是否允许用户临时 key。
3. 不同后端的错误信息应映射为结构化错误码。

**当前落地**：
- `pdf_translate.translators.registry` 统一声明 `echo`、`deepseek`、`openai`、`ollama`、`deepl`、`hybrid` 的 id、展示名称、别名、配置字段、默认启用状态、自定义 API 支持和段尾顺延能力。
- `build_translator()`、CLI help、成本模型名、`_translator_supports_deferral()`、Web 用户后端列表和管理端设置均从 registry 读取后端定义。
- 管理端保存 `enabled_backends` 时会归一化别名并拒绝未知后端；用户端自定义 API 的可选后端由服务端返回，避免前后端口径不一致。

**验收**：
- 新增一个后端时不需要分别改 4-5 个位置。
- 前端展示、后端校验和实际构造器行为一致。

### Web 任务状态

**适用范围**：`server/jobs.py`、`server/routes_web.py`、`server/database.py`、Web 用户端和管理端任务列表。

**规则**：
1. Web 任务运行态以 `JobRegistry` 和 `data/web_jobs/<job_id>/web_status.json` 为主源。
2. `web_status.json` 只记录任务状态、阶段、进度、错误、运行时间、页面统计、任务所有者、原始文件名和可安全恢复的运行参数等运行态快照。
3. SQLite `jobs_meta` 只负责用户归属、任务列表、收藏列表和列表时间，不应承载运行进度。
4. 管线 `output/state.json` 只负责 chunk 断点续跑，不等同于 Web 任务状态。
5. 服务启动恢复任务时，应以 `web_status.json` 所在目录作为 `work_dir`，不要信任快照内可被篡改的路径字段。
6. 用户任务列表和管理员任务列表应通过 `JobRegistry.merge_status_into_rows()` 合并运行态字段，同时保留 SQLite 的 `created_at` 列表时间。
7. 任务列表和任务详情应复用同一份产物摘要，至少暴露原 PDF、译文 Markdown、双语 HTML、结构化 PDF 译文、表格合并候选确认清单、表格结构确认发布报告、确认后的表格重建副本、局部修复补丁审核报告、局部修复效果对比报告、局部修复发布确认报告、人工确认修复发布稿、局部修复回滚演练报告、回滚演练稿、zip 打包可用性和一致性 warning。
8. 历史任务清理必须先巡检再执行；管理员清理只能删除数据库缺工作目录的行，以及没有数据库索引的任务目录，运行中任务必须跳过。
9. CLI、用户详情和管理员列表应复用同一套诊断字段；`output/state.json` 只说明 chunk 断点，必须以 `pipeline_state_status`、`pipeline_resume_ready` 和 `job_recovery_status` 的形式进入恢复审计。
10. 任务生命周期审计应覆盖提交、启动、取消请求、取消终态、完成、失败和服务启动恢复扫描；审计详情应保留状态、阶段、耗时、错误码、翻译模式、并发数、原始文件名和关键产物路径。审计写入失败不应阻断任务主流程。
11. 自动恢复默认关闭；只有显式配置后才允许把重启前的活动任务重新入队，且不得恢复依赖用户临时 API Key 的自定义接口任务。

**当前落地**：
- `JobRecord.to_status_dict()` / `JobRecord.from_status_dict()` 统一任务状态快照结构，并写入 `schema_version: web-job-status-v1`。
- `_persist()` 通过临时文件加 `os.replace()` 原子更新 `web_status.json`，避免写入半截 JSON。
- `web_status.json` 现在会持久化 `backend`、`tail_fallback`、`pages_per_chunk`、`overlap_pages`、`max_chunks` 和 `use_custom_api`，用于服务重启后的受控恢复判断；这些参数不从审计日志反推。
- `/api/user/jobs` 和 `/api/admin/jobs` 已在返回列表时合并运行态字段；历史缺失状态快照的行会返回 `status_available: false`。
- `artifact_fields_for_record()` 会计算 `input_pdf_ready`、`partial_output_ready`、`bilingual_html_ready`、`translated_pdf_ready`、`table_merged_cell_review_ready`、`table_merged_cell_review_required_count`、`table_merged_cell_review_human_reviewed_count`、`repair_patch_review_ready`、`repair_effectiveness_report_ready`、`repair_effectiveness_status`、`repair_effectiveness_issue_reduction_rate`、`repair_effectiveness_issue_delta`、`repair_publish_report_ready`、`repair_publish_confirmed`、`repair_publish_published`、`repair_publish_open_issue_count`、`repair_published_full_ready`、`repair_rollback_report_ready`、`repair_rollback_available`、`repair_rollback_confirmed`、`repair_rollback_applied`、`repair_rollback_status`、`repair_rollback_matches_original`、`repair_rollback_full_ready`、`bundle_zip_ready`、`artifact_consistent`、`artifact_consistency_status` 和 `artifact_warnings`；管理端任务表会展示产物状态，并禁用不可下载的产物按钮。
- 正式稿层会额外暴露 `repair_formal_replace_report_ready`、`repair_formal_replace_available`、`repair_formal_replace_replaced`、`repair_formal_replace_status`、`repair_formal_full_ready`、`repair_formal_backup_full_ready`、`repair_formal_rollback_report_ready`、`repair_formal_rollback_applied`、`repair_formal_rollback_status` 和 `repair_formal_active_before_rollback_full_ready`，用于区分发布副本、用户可见正式稿、修复前备份和回滚前修复稿。
- 用户端提供 `/api/jobs/{job_id}/download/table-merged-cell-review.md`、`/api/jobs/{job_id}/download/table-structure-publish.md`、`/api/jobs/{job_id}/download/table-reconstruction-confirmed.json`、`/api/jobs/{job_id}/download/repair-patch-review.md`、`/api/jobs/{job_id}/download/repair-effectiveness.md`、`/api/jobs/{job_id}/download/repair-publish.md`、`/api/jobs/{job_id}/download/published-full.md`、`/api/jobs/{job_id}/download/repair-rollback.md` 和 `/api/jobs/{job_id}/download/rollback-full.md`，管理端产物下载支持 `kind=table_merged_cell_review`、`kind=table_structure_publish`、`kind=table_reconstruction_confirmed`、`kind=repair_patch_review`、`kind=repair_effectiveness`、`kind=repair_publish`、`kind=repair_published_full`、`kind=repair_rollback` 与 `kind=repair_rollback_full`，用于审阅表格合并候选确认清单、表格结构发布确认报告、确认后的表格重建副本、局部修复补丁级审核报告、修复效果对比报告、发布确认报告、人工确认修复发布稿、回滚演练报告和回滚演练稿。
- 正式稿相关下载新增 `/api/jobs/{job_id}/download/repair-formal-replace.md`、`/api/jobs/{job_id}/download/repair-formal-rollback.md`、`/api/jobs/{job_id}/download/formal-full.md` 和 `/api/jobs/{job_id}/download/formal-before-repair-full.md`；管理端产物下载支持 `kind=repair_formal_replace`、`kind=repair_formal_rollback`、`kind=repair_formal_full`、`kind=repair_formal_backup_full` 与 `kind=repair_formal_active_before_rollback_full`。
- 用户端和管理端可调用 `GET /api/jobs/{job_id}/table-merged-cell-review` 读取表格合并候选确认清单，并通过 `POST /api/jobs/{job_id}/table-merged-cell-review/{review_id}` 写回单条 `confirm`、`reject`、`needs_revision` 或 `clear` 人工决策。该写回只更新 `table_merged_cell_review.json/md`、同步刷新 `table_structure_publish.json/md` 的待确认状态和任务状态摘要，不反向覆盖 `table_reconstruction.json`。
- 用户端和管理端可调用 `POST /api/jobs/{job_id}/table-merged-cell-review/batch` 批量写回多个 `review_id` 的同一类人工决策。后端会先校验全部目标 ID，存在缺失或非法决策时不做半更新；成功后同样刷新 `table_structure_publish.json/md` 并写入审计日志。
- 用户端和管理端可调用 `GET /api/jobs/{job_id}/table-merged-cell-review/{review_id}/preview.png` 渲染单条合并候选所在页预览图。该接口只从当前任务 `input.pdf` 渲染页面，并按 `bbox_evidence.span_bbox`、`evidence_bbox`、`candidate_bbox` 的顺序画框，不接收任意文件路径。
- 前端共用 `TableMergedCellReviewModal` 展示候选列表、位置、跨度、证据等级、当前状态、文本片段、备注和四类决策按钮，替代手工输入 `review_id` 的 prompt 流程。该组件已支持状态筛选、证据筛选、关键词搜索、当前筛选勾选、批量备注、批量确认/拒绝/复查/清空和单条候选 bbox 页面预览；键盘流转和多人审核记录仍待完善。
- 用户端和管理端可调用 `GET /api/jobs/{job_id}/table-structure-publish` 查看表格结构发布状态，并通过 `POST /api/jobs/{job_id}/table-structure-publish/confirm` 显式生成 `table_reconstruction_confirmed.json`。只有 `table_merged_cell_review.summary.review_required_count == 0` 时才会发布；发布时会把人工确认后的合并单元格候选物化为 `table_structure_patches`，并在确认副本的 cells 中保留可回滚的 span/覆盖关系元数据。发布后如已有译文 chunk 产物，会用确认视图刷新 `translated_full.pdf` 与 `translated_pdf_report.json`。若人工审核又退回为 `needs_revision` 或 pending，旧确认副本会失效删除，避免下载陈旧表格结构。
- 用户端和管理端可调用 `GET /api/jobs/{job_id}/repair-patch-review` 读取补丁审核清单，并通过 `POST /api/jobs/{job_id}/repair-patch-review/{review_id}` 写回单条人工决策；`POST /api/jobs/{job_id}/repair-patch-review/batch` 可批量写回多个 `review_id` 的同一类决策。批量接口会先校验所有目标 ID 和决策值，失败时不做半更新；成功后重写 `repair_patch_review.json/md`，把人工决策、审核人、时间和备注写入报告，并记录审计日志。`POST /api/jobs/{job_id}/repair-publish/confirm` 显式确认修复发布时会优先保留既有 `repair_patch_review.json/md` 的人工决策；若仍有 `publish_blocking_count > 0`，则写入 `repair_publish.json/md` 的 `blocked_patch_review` 状态，不生成 `output/published_full.md`。`POST /api/jobs/{job_id}/repair-rollback/confirm` 只在已发布副本存在时生成 `output/rollback_full.md` 回滚演练稿，并刷新 `repair_rollback.json/md`；该动作不覆盖 `published_full.md`，用于验证原始译文基线仍可恢复。
- 前端共用 `RepairPatchReviewModal` 展示局部修复补丁列表、定位、风险、有效结论、候选片段、差异预览、备注和四类决策按钮，替代用户端/管理端手工输入 `review_id` 的 prompt 流程。该组件支持发布阻断/未处理/可发布/已拒绝/需复查筛选、风险筛选、关键词搜索、当前筛选勾选、批量备注、批量通过/拒绝/复查/清空和补丁详情侧栏；详情侧栏读取后端 `diff_preview`，展示源文摘要、当前译文、修复候选、合并后片段和统一 diff，旧数据可回退显示 `result_excerpt`。
- 用户端和管理端可调用 `POST /api/jobs/{job_id}/repair-formal-replace/confirm` 把已发布的 `published_full.md` 显式提升为 `formal_full.md`，并在首次替换前备份 `formal_full.before_repair.md`；`POST /api/jobs/{job_id}/repair-formal-rollback/confirm` 会先保存当前修复稿到 `formal_full.repair_applied.md`，再从备份恢复 `formal_full.md`。这两个动作不覆盖 `translated_full.md` 或 `published_full.md`。
- `storage_drift()` 会对比 SQLite 索引和 `web_jobs/` 目录，输出数据库缺目录、目录缺数据库索引和运行中任务；`/api/admin/jobs/reconcile` 支持 `apply=false` 巡检和 `apply=true` 清理，并在执行清理时写入审计日志。
- `pipeline_state_fields_for_record()` 会读取 `output/state.json`、`output/chunks_manifest.json` 和 `output/chunks/`，输出已完成 chunk 数、待处理 chunk、续跑可用性、恢复状态和诊断 warning。
- `pdf-translate web-status` 会读取同一份 `web_status.json` 与管线 state 诊断，供 CLI/脚本侧查看 Web 任务恢复状态；`/api/admin/jobs/hydration-report` 会暴露本次启动恢复报告与存储漂移摘要。
- `requeue_recovered_jobs()` 会评估重启恢复出的 `queued` / `running` 任务，只有输入 PDF 存在、无取消标记、恢复参数完整且不使用用户临时自定义 API 时才会重新入队。`PDF_TRANSLATE_JOB_AUTO_RESUME=off|safe|all` 控制启动时是否自动重新入队，默认 `off`；`PDF_TRANSLATE_JOB_AUTO_RESUME_MAX` 控制单次最多恢复数量，默认 2，上限 32。管理端可通过 `POST /api/admin/jobs/{job_id}/requeue` 手动恢复单个符合条件的任务。
- 任务生命周期审计已记录 `job_submit`、`job_started`、`job_cancel_requested`、`job_done`、`job_cancelled`、`job_error`、`job_hydration_report` 和 `job_recovery_requeued`；完成、取消和失败终态会携带终态、阶段、耗时、错误结构、翻译模式、并发数和产物路径，服务启动会把恢复扫描报告落入审计日志。

**验收**：
- `tests/test_job_status.py` 覆盖状态快照 schema、重启恢复、恢复报告、恢复重新入队、持久化运行参数、路径边界、列表合并、缺失状态快照、完成态缺失译文的产物异常、管线 state 诊断、CLI `web-status`、存储漂移识别、路径穿越清理保护，以及启动、取消请求、取消终态、失败终态、启动恢复扫描和恢复重新入队审计。

### 结构化错误码

**适用范围**：`pdf_translate/error_codes.py`、`translators/`、`server/jobs.py`、`run_metrics.py`、`qa/metrics.py`、`frontend/src/views/UserView.vue`。

**规则**：
1. 底层异常应先映射为 `ErrorInfo`，再进入任务状态、日志和指标；不要只向上抛一段不可分类的字符串。
2. `ErrorInfo` 字段包含 `code`、`category`、`retryable`、`user_message`、`next_step`、`source`、`http_status` 和 `exception_type`。
3. 当前稳定错误码包括 `CONFIG_MISSING_API_KEY`、`CONFIG_INVALID_BACKEND`、`CONFIG_INVALID_OPTION`、`HTTP_RATE_LIMIT`、`HTTP_TIMEOUT`、`HTTP_AUTH_ERROR`、`HTTP_CLIENT_ERROR`、`HTTP_SERVER_ERROR`、`HTTP_NETWORK_ERROR`、`API_RESPONSE_INVALID`、`PDF_PARSE_ERROR`、`TASK_CANCELLED`、`PIPELINE_ERROR` 和 `UNKNOWN_ERROR`。
4. Web 任务状态保留旧字段 `error`，同时新增 `error_code`、`error_category`、`error_retryable`、`error_next_step`、`error_source` 和 `error_http_status`，供前端显示下一步建议。
5. 前端应优先显示 `user_message` / `detail` / `message`，同时展示 `next_step` / `error_next_step` 与 `code` / `error_code`，不要把结构化 JSON 原样弹给用户。
6. `run_metrics.json.summary` 和 `experiment_metrics.json.breakdowns` 必须保留错误码/错误类别分布，便于批量实验统计失败原因。

**当前落地**：
- `pdf_translate.error_codes.PdfTranslateError` 是结构化异常入口，`error_info_from_exception()` 负责把 HTTP 状态、超时、网络错误、PDF 解析错误、取消和配置错误映射为稳定错误码。
- `call_with_http_retry()` 的捕获事件会写入 `error_code`、`error_category` 和 `error_retryable`；重试耗尽或非重试 HTTP 错误会抛出 `PdfTranslateError`。
- `build_translator()` 对缺失 API Key 和未知后端抛出结构化配置错误；`server/jobs.py` 会把失败或取消写入 `web_status.json` 并通过 `/api/jobs/{job_id}` 暴露。
- `RunMetricsRecorder.stage()` 会在阶段异常中记录错误码，`build_run_metrics()` 会汇总 `error_code_counts` 和 `error_category_counts`，并由 `experiment_metrics.json` 回链。
- `frontend/src/views/UserView.vue` 会格式化 FastAPI `detail` 对象和扁平错误字段，提交、取消、下载失败提示以及任务卡片可展示用户消息、下一步建议、错误码和 HTTP 状态。

### 记忆与术语

**适用范围**：`memory_store.py`、译前巡视、精品翻译、未来术语确认台。

**规则**：
1. 术语应区分候选、已确认、已否决、冲突待审；当前新增术语默认写为 `candidate`，人工审核写回后可变为 `confirmed` 或 `rejected`。
2. 术语应保留来源、页码、章节、结构类型和更新时间。
3. 当前块只注入相关术语，避免全量术语挤占上下文窗口。
4. 模型草拟术语不能默认等同于人工确认术语。

**原因**：术语一致性是学术翻译质量的主要杠杆，也是专利方向中的动态记忆反馈证据。

**验收**：
- 同一英文多译、同一中文多源能进入 `pending_review.json` 或等价待审结构。
- `MemoryStore.apply_glossary_review_decision()` 是当前术语审核后端入口：确认候选会更新同一英文术语的活动中文译名，否决候选只关闭待审项；两类动作都应记录审核人、时间、备注，可选记录置信度和章节作用域。
- 术语导入导出不破坏现有 `glossary.json`。

**当前落地**：
- `merge_glossary_terms_from_survey()` 新增术语时会写入 `status: candidate`。
- 同一英文术语出现不同中文译名时，不静默覆盖原术语，而是写入 `pending_review.json` 的 `glossary_conflict`。
- 多个英文术语共享同一中文译名时，会写入 `shared_translation_review`，供人工确认是否为同义术语。
- 译后 QA 会读取 `pending_review.json`，在相关块中报告 `glossary_translation_conflict`，修复计划会映射为 `review_glossary_conflict`。

### PDF 结构层

**适用范围**：未来 `extractors/`、`DocumentIR`、结构感知分块、QA。

**规则**：
1. 不直接把复杂 PDF 线性化为不可区分的普通文本。
2. 表格、图注、脚注、注释、人名、机构名、公式、标题、参考文献、页眉页脚应有类型边界。
3. 表格必须保留行列关系、表头、单位、脚注和显著性符号，不能被当作普通段落拼接。
4. 图注、脚注、页下注释和人名机构信息应能回到所属图表、段落或页面。
5. 分块不得切断表格、图注、公式环境和跨页未完段落。
6. 数字、公式编号、引用编号、单位和表格维度应作为结构不变量进入 QA。
7. 结构分段必须记录预算来源、切分原因、预算压力、估算 token 和结构关系保护，不能只产出不可解释的 chunk。

**原因**：当前最大质量风险不是“模型不会翻译”，而是模型看到的输入结构已经被破坏。

**验收**：
- 对表格密集论文，输出能保留行列关系或至少给出风险标记。
- QA 报告能指出数字、引用、公式和表格维度异常。

**当前落地**：
- `pdf_translate.extractors.document_ir.extract_document_ir()` 会生成 `output/document_ir.json`。
- `pdf_translate.chunkers.structure.build_structure_chunks()` 会生成 `output/structure_chunks_manifest.json`，记录 `block_ids`、`boundary_fragment_ids`、`structural_relation_ids`、`approx_tokens` 和 `budget` 元数据，并在页面边界高风险或图注/表注/脚注父子关系相邻时保护结构关系不被目标字符数或普通页数阈值提前拆开。
- 表格块会在 `meta.table` 中记录行列、表头、数字 token、confidence 和 warning；结构分段会把表格优先渲染成 Markdown table。
- 图注、表注和脚注会按同页优先、一页跨页回退的规则写入 `parent_id` 与 `meta.parent_relation`；跨页归属会记录 `parent_page_no`、`parent_page_gap`、`cross_page_parent` 和 `cross_page_parent_attempted`，紧跟表格的脚注会标记为 `footnote_for_table`，无法归属的内容会写入 `meta.parent_warning`。
- 文本块会用本地规则提取人名、机构、模型/数据集名和缩写候选，写入 `meta.entities`；这些结果只表示候选实体，不等同于人工确认术语。
- `pdf_translate.qa.structure.write_structure_qa()` 会生成 `output/structure_qa.json`，记录表格维度、续表风险、图注/表注/表格脚注/脚注关系、实体候选、孤立关系统计、跨页关系成功/孤立统计、页级风险、页面边界碎片和结构块统计。
- `pdf_translate.qa.table_reconstruction.write_table_reconstruction_report()` 会生成 `output/table_reconstruction.json`，按源表格输出单元格坐标、表头、行头、数字/单位/显著性 token、表注/脚注表级绑定、`footnote_bindings` 脚注单元格级绑定证据、ragged 行/空单元格统计、`merged_cell_candidates` 合并单元格候选证据和续表关系；来自 canonical `block.meta.table.merged_cell_candidates` 的 OCR/meta 候选会先兼容规范化，再与本地行宽/空单元格推断候选按几何范围合并去重，避免同一合并位置重复计数；跨页续表会额外进入 `continued_table_groups`，记录续表片段链、兼容性检查、合并/拒绝状态、拒绝/警告原因分布、合并后行列视图、重复表头跳过数、组级合并单元格候选和组级可重建状态，作为表格保护式翻译、PDF 输出和专利实验的前置证据。
- 合并单元格候选必须继续按“候选”处理。`candidate_status` 只允许表示 `candidate`、`visually_supported`、`human_confirmed` 或 `rejected`；代码端只能根据外部 span bbox 或真实 OCR cell bbox 提升到 `visually_supported`，不能自动写出 `human_confirmed`。`visual_evidence_level` 用于区分 `none`、`estimated_bbox`、`ocr_cell_bbox`、`visual_span_bbox` 和未来人工确认；`bbox_evidence.status` 用于区分 `missing`、`estimated`、`ocr_reported`、`span_reported` 和未来 `manual_verified`。本地区域均分 bbox 只能产生 `estimated_bbox` 证据，不得被解释为视觉确认。
- `pdf_translate.qa.table_reconstruction.write_table_merged_cell_review()` 会在 OCR 晋级后的表格重建基础上生成 `output/table_merged_cell_review.json` 和 `output/table_merged_cell_review.md`。该报告把每个合并候选整理为稳定 `review_id`、页码、表格、锚点、跨度、候选状态、证据等级、默认决策和 `human_decision` / `human_comment` / `reviewed_by` / `reviewed_at` 占位；`write_table_merged_cell_review_decision()` 可写回 `confirm`、`reject`、`needs_revision` 或 `clear`，并重算待确认、已复核、已确认、已拒绝和需修改数量。它是人工确认清单，不会把 `visually_supported` 自动变成 `human_confirmed`，也不会直接改写表格结构。
- `pdf_translate.qa.table_reconstruction.write_table_structure_publish()` 会根据 `table_merged_cell_review.json` 生成 `output/table_structure_publish.json` 与 `output/table_structure_publish.md`。默认状态为 `pending_confirmation`；用户显式确认且没有待复核/需修改候选时，才写出 `output/table_reconstruction_confirmed.json`。确认副本会新增顶层 `table_structure_patches`，并在对应表格下写入 `structure_patches`、`table_structure_patch_count` 和单元格级 `effective_row_span` / `effective_column_span` / `covered_by_merged_cell_patch_id` 元数据；这些字段表示可回滚结构补丁，不会删除原始单元格文本。`load_preferred_table_reconstruction()` 会在后半段 QA、修复后 QA 和 PDF 导出中优先读取确认副本，并把下游可见的合并候选收敛为 `confirmed_merged_cell_candidates`，同时保留结构补丁统计；确认副本不存在或无效时回退原始 `table_reconstruction.json`。该确认副本不覆盖原始 `table_reconstruction.json`；若发布被阻断，会清理旧确认副本，避免陈旧结构被误用。
- `pdf_translate.qa.table_reconstruction.build_structure_translation_hints()` 会把当前 chunk 覆盖的表格证据、图注/脚注父子关系和实体候选压缩为 `TranslationRequest.structure_hints`，OpenAI/DeepSeek 兼容后端会以 `v5`「结构保护提示」注入模型消息，要求保留 Markdown 表格行列、表头/行头关系、脚注单元格绑定、续表合并关系、疑似合并单元格候选、图注/脚注归属和专名候选；合并单元格候选必须以“未确认结构风险”处理，实体候选只作弱提示，不得提示模型自行声称已经完成合并、重排或人工确认术语。
- `pdf_translate.qa.table_reconstruction.write_structure_hints_manifest()` 会生成 `output/structure_hints_manifest.json`，把每个 chunk 实际注入翻译请求的 `structure_hints` 固化为审计清单，记录覆盖页码、块 ID、表格 ID、续表组 ID、提示文本、提示长度、空提示 chunk 数、脚注绑定数、结构关系提示数、跨页关系提示数、实体提示数、锁定 token 数以及合并单元格候选类型/原因分布；串行和并行翻译路径都从该清单复用提示文本，避免“记录的提示”和“实际发送的提示”不一致。
- `pdf_translate.qa.chunk_boundary.write_chunk_boundary_qa()` 会生成 `output/chunk_boundary_qa.json`，对照结构 QA 的 `page_boundary_fragments` 与当前实际 chunks，统计高风险页面边界是被保护、同块覆盖还是被切开，并汇总 `budget_split_reason_counts`、`budget_pressure_counts`、`budget_overflow_chunk_count` 和 `structural_relation_protected_count`。
- `pdf_translate.qa.chunk_boundary.write_chunk_strategy_comparison()` 会生成 `output/chunk_strategy_comparison.json`，同一份文档下同时评估固定页分段和结构分段的边界切开率、保护率和相对基线降幅。
- `pdf_translate.qa.translation.write_translation_qa()` 会生成 `output/qa_report.json` 和 `output/qa_report.md`，检查数字、引用、图表编号、公式符号、公式编号、变量/统计表达不变量、Markdown 表格形状、DocumentIR 表格形状、表格单元格锁定 token、术语一致性、术语冲突、实体候选保留、重复段落和英文残留；术语一致性检查读取当前任务 `memory/glossary.json` 和 `memory/pending_review.json`，表格结构检查读取当前任务 DocumentIR 与 `table_reconstruction.json`，表格单元格 token 异常会在 confirmed 视图下为当前 cell 收窄匹配 `matched_structure_patches`，公式不变量检查会把 `formula_mismatch` 交给局部修复计划生成公式邻近段落重写请求。
- `pdf_translate.qa.repair.write_repair_plan()` 会生成 `output/repair_plan.json` 和 `output/repair_plan.md`，把 QA 问题映射成块级、段落级、表格级、表格单元格级、实体级或术语级修复建议，术语缺失会映射为带术语锁定要求的块级重译，实体缺失会映射为带实体锁定要求的块级重译，表格单元格 token 缺失会映射为 `repair_table_cell_tokens`，术语冲突会映射为人工术语复核。
- `pdf_translate.qa.repair.write_repair_requests()` 会生成 `output/repair_requests.json` 和 `output/repair_requests.md`，把修复计划转成可交给翻译后端或人工复核的请求信封，包含源文片段、当前译文片段、锁定 token、QA 证据、局部修复指令、目标表格/单元格合并定位和后端 payload。若 QA evidence 含当前 cell 匹配到的 confirmed `structure_patches`，请求会把它提升为 `merge_target.structure_patch_context`、`structure_patch_evidence_count` 与 `structure_patch_source_review_ids`，后续验证、合并、补丁审核和实验指标都会保留该定位依据；该字段只作为定位证据，不替代锁定 token 与表格形状验证门禁。
- `pdf_translate.qa.repair.write_repair_results()` 会生成 `output/repair_results.json` 和 `output/repair_results.md`；默认记录每条请求因执行开关关闭而跳过，CLI 显式传入 `--execute-repairs` 后才会调用翻译后端，并把修复片段写入 `output/repairs/`，不直接覆盖原译文 chunk。
- `pdf_translate.qa.repair.write_repair_validation()` 会生成 `output/repair_validation.json` 和 `output/repair_validation.md`，对成功生成的候选修复片段做本地不变量验证，当前重点检查锁定 token 保留与表格 Markdown 形状；表格单元格 token 修复会从 QA cell 证据携带源表格行列形状，因此候选片段即使补回 token 但破坏表格形状也会被门禁拦下。
- `pdf_translate.qa.repair.write_repair_merge()` 会生成 `output/repair_merge.json`、`output/repair_merge.md`、`output/repaired_chunks/` 和 `output/repaired_full.md`；仅把验证通过且可安全定位的候选修复写入独立修复副本，当前支持 chunk 级替换、按 QA 证据定位目标 Markdown 表格替换和首个 Markdown 表格兜底替换，不覆盖原始 `output/chunks/`。
- 管线会对 `output/repaired_chunks/` 再运行一轮译后 QA，生成 `output/repair_merge_qa.json` 和 `output/repair_merge_qa.md`，用于观察合并后 issue 数和表格 token 异常是否下降；随后 `pdf_translate.qa.repair_effectiveness.write_repair_effectiveness()` 会生成 `output/repair_effectiveness.json` 和 `output/repair_effectiveness.md`，按问题类型、严重程度和 chunk 状态对比修复前后差异，记录已解决、持续存在、新增和回归问题。Web 任务摘要会读取 `repair_effectiveness.json.summary`，暴露修复前后问题数、问题减少率、已解决/新增/回归数量和状态；用户端与管理端均可下载 `repair_effectiveness.md`。
- `pdf_translate.qa.repair.write_repair_patch_review()` 会在 `repair_merge` 之后、`repair_publish` 之前生成 `output/repair_patch_review.json` 和 `output/repair_patch_review.md`，按补丁列出默认审核结论、有效审核结论、风险等级、是否阻断发布、合并目标、结果片段、`diff_preview` 和 `human_decision` / `human_comment` / `reviewed_by` / `reviewed_at` 字段；`diff_preview` 来自 `repair_merge`，包含源文摘要、当前译文、修复候选、合并后片段和统一 diff，表格补丁优先截取目标 Markdown 表格；`write_repair_patch_review_decision()` 可写回 `approve`、`reject`、`needs_revision` 或 `clear`，并重算 `publish_blocking_count`。

**QA 边界**：
- 当前译后 QA 是纯规则检查，不调用模型，不直接修改译文；图注/脚注关系锚点和表格脚注绑定也会作为译后结构保真证据进入 QA、修复计划和实验指标。
- QA JSON 面向实验统计，QA Markdown 面向导师评审和人工复核。
- 结构边界检测会在翻译前检测相邻页面首尾块的断句、续表和接续风险，QA 输出 `page_boundary_fragments` 与 `table_continuations`，结构分段清单输出 `boundary_fragment_ids`、`structural_relation_ids` 和 `budget`，分段边界 QA 输出 `split_boundary_count`、`protected_boundary_count`、`high_risk_split_count`、`budget_split_reason_counts`、`budget_pressure_counts`、`budget_overflow_chunk_count` 和 `structural_relation_protected_count`，策略对比输出 `active_split_reduction_vs_baseline`。
- 关系归属检测会在 `relationships` 中记录 caption/footnote 的父块、关系类型、父页码、页距、跨页标记和孤立 warning，用于后续图文重排、PDF 输出和结构保持率统计；跨页回退当前只允许相邻 1 页，避免误把远距离图表强行绑定。
- 表格重建报告会在翻译前保留表格单元格上下文、表级脚注、脚注单元格级绑定证据和合并单元格候选证据，跨页续表会先经兼容性闸门再生成组级合并视图，并把拒绝原因和警告原因同时聚合为原始分布与稳定类别分布；译后 QA 会把锁定 token 缺失定位到表格单元格坐标，并在确认视图存在时携带已确认合并候选与当前 cell 匹配到的结构补丁作为 evidence。合并单元格已具备“候选识别 -> 人工确认 -> 独立确认副本 -> 可回滚结构补丁 -> 双语 HTML/结构化 PDF 基础渲染 -> 单元格级修复定位依据”的闭环；高精度识别、复杂原位重排和正式替换执行仍需后续实现。
- 实体候选检测会在 `entities` 中记录块 ID、页码、实体类型、文本、置信度和规则来源，用于后续术语确认台、实体一致性 QA 和专利实验统计。
- 译后实体 QA 会检查候选实体是否在译文中保留；当前只报告缺失候选，不判断中文译名是否正确。
- 译后表格 QA 会优先用 Markdown 表格形状比对；当源 chunk 不是 Markdown 表格但 DocumentIR 中存在表格块时，会按页码或结构 chunk 的 `block_ids` 读取源表格行列数，发现译文没有保留 Markdown 表格或行列不一致时输出 `table_shape_mismatch`；当目标表格对应单元格缺失数字、单位或显著性符号等锁定 token 时输出 `table_cell_token_mismatch`；当表格脚注标记在源单元格与目标单元格之间断开时输出 `table_footnote_binding_mismatch`。
- 译后结构关系 QA 会读取 DocumentIR 中 caption、footnote、table note 等父子关系，检查译文是否保留 `Table 1`、`Figure 2`、`*` 等稳定锚点；缺失时输出 `caption_or_footnote_relation_mismatch`，修复计划会将其映射为带锁定锚点的结构关系重写请求。
- 修复计划会读取 QA JSON 中的异常块，给出优先级、修复范围、执行器和证据；修复请求会继续读取源 chunk 与当前译文，生成可由后端执行的局部修复 payload，并为表格/单元格问题记录目标表格索引、表格 ID、目标单元格摘要和可选的 confirmed 结构补丁上下文；修复结果只记录执行状态与候选修复片段；修复验证只判断候选片段是否满足本地不变量，结构补丁上下文只作为定位证据随 `merge_target` 透传；修复合并只写独立副本和合并报告；补丁审核报告负责把合并结果整理为可人工逐项复核并写回决策的列表；修复发布确认只在显式 `--publish-repairs` 或 Web 确认且补丁审核无阻断项后生成 `published_full.md` 发布副本，不覆盖原始译文；回滚演练只在显式 `--rollback-repairs` 或 Web 确认后生成 `rollback_full.md`，用于验证原始译文基线可恢复。
- 正式替换层在发布副本之后单独执行：`published_full.md` 是审核通过的发布副本，`formal_full.md` 是用户可见正式稿，`formal_full.before_repair.md` 是覆盖前备份，`formal_full.repair_applied.md` 是回滚前修复稿快照。该层仍以显式确认触发，避免自动修复结果直接污染正式输出。
- 实验指标不会替代明细文件；它只聚合可复现实验所需的摘要、比例和证据文件路径，后续批量样本对比应读取该文件并回链到结构 QA、结构提示清单、分段边界 QA、译后 QA、修复计划、修复请求、修复结果、修复验证、修复合并和合并后 QA 等明细 JSON。
- 成本估算只在命中成本画像时给出金额；未配置画像时仍保留 token、字符、翻译请求次数、HTTP 尝试次数和可计费请求次数，不得把未配置价格的 `0` 误读为真实免费。失败 HTTP 尝试的 token/字符是否计费取决于服务商账单规则，当前只把 `per_request` 按尝试次数计入，失败尝试 token 成本仍需用真实账单校准。
- 术语一致性 QA 目前可检查源文出现的英文术语是否使用术语库指定中文译名，并能暴露待审术语冲突；已确认术语的置信度和章节作用域可随缺失术语证据透出。可编辑确认台、批量编辑和确认后重译触发仍待完善。
- 当前修复执行不会自动覆盖正式译文；`repair_patch_review.json` / `repair_patch_review.md` 负责记录补丁级默认结论、有效结论、风险、差异预览、人工决策、人工备注和发布阻断状态，`repair_effectiveness.json` / `repair_effectiveness.md` 负责记录修复前后 QA 问题减少、新增、持续存在和回归情况，`repair_publish.json` / `repair_publish.md` 负责记录发布确认请求状态、发布副本路径、开放合并问题数、补丁审核阻断数和回滚基线，其中 `confirmed` 表示用户显式请求发布，`published` 才表示发布副本已经生成。Web 确认发布只生成 `published_full.md` 审核副本，不替换原始 `translated_full.md`；若补丁审核仍有阻断项，则只记录 `blocked_patch_review`，不生成发布副本。`repair_rollback.json` / `repair_rollback.md` 负责记录回滚演练状态，确认后生成 `rollback_full.md` 并校验其与原始译文一致。`repair_formal_replace.json` / `repair_formal_replace.md` 负责记录正式替换状态，确认后生成或更新 `formal_full.md` 并保留替换前备份；`repair_formal_rollback.json` / `repair_formal_rollback.md` 负责记录正式回滚状态，确认后恢复 `formal_full.md` 并保留回滚前修复稿快照。后续自动修复器应继续消费 `repair_results.json`、`repair_validation.json`、`repair_merge.json`、`repair_patch_review.json`、`repair_effectiveness.json`、`repair_publish.json`、`repair_rollback.json`、`repair_formal_replace.json`、`repair_formal_rollback.json`、`repair_merge_qa.json` 与 `output/repairs/`，下一步重点是真实模型跑批效果对比和复杂 PDF 正式稿重排。

**阅读输出当前落地**：
- `pdf_translate.exporters.bilingual_html.write_bilingual_html()` 会生成 `output/bilingual.html`。
- 双语 HTML 按 chunk 展示原文与译文，并在同页展示 QA 问题和局部修复建议。
- 双语 HTML 会读取下游优先表格重建视图；只有 `table_structure_source=confirmed` 或带确认 schema 的表格确认副本，才会把 `structure_patches` 叠加到译文侧 Markdown 表格，生成 `rowspan` / `colspan` 并跳过 covered cell。未确认 source 视图中的候选或补丁字段不得改变阅读产物。
- `pdf_translate.exporters.translated_pdf.write_translated_pdf()` 会生成 `output/translated_full.pdf` 和 `output/translated_pdf_report.json`，按 chunk 重排标题、段落、Markdown 表格、QA 问题和修复建议；在 confirmed 表格视图下会按 `structure_patches` 对译文 Markdown 表格进行基础 `colspan` / `rowspan` 绘制并跳过 covered cell，同时把页数、chunk 数、表格数、QA 数、caption/footnote 跨页关系、表格脚注单元格绑定、续表组、结构关系、表格重建来源、已确认合并候选引用数和 `table_structure_patch_rendered_count` 写入报告。
- 当前 HTML 和 PDF 都是评审/对照阅读产物，不追求 PDF 原坐标回填；结构化 PDF 已能消费确认结构补丁做基础表格合并显示，后续仍应继续从结构化块和修复后译文重新排版，补齐复杂图表、脚注、跨页表格分页联动和页码关系。

### 图像与 OCR

**适用范围**：未来 `extractors/`、OCR 适配器、页面截图、VLM 路由、图文 QA。

**规则**：
1. 不能只依赖“本页文本数量低于平均值”判断是否需要图像模型；该规则只能作为低成本启发。
2. 页面应保留文本层、图片区域、页面截图、图注和 OCR 置信度等信息。
3. 扫描页、图片型表格、公式密集页和图注密集页应进入风险列表。
4. 本地 OCR 优先服务于结构识别和图文边界恢复，不把图片翻译作为第一目标。

**原因**：图像能力的价值在于避免漏掉扫描页、图表和图注，而不是简单提高模型调用次数。

**验收**：
- 低文本页能区分扫描页、图片页、公式页和普通空白页。
- 开启 OCR/VLM 后，产物能说明哪些页面调用了图像处理，原因是什么，并能回链到对应页面预览。

**当前落地**：
- 管线会生成 `output/vision_route.json`，基于文本字符数、文本区域比例、图像数量、图像区域比例、图注/表格/公式块和页级 warning 给出路由建议。
- 非 `text_only` 页面会生成 `output/vision_pages/` 下的页面预览图，并在 `vision_route.json.pages[].evidence` 中记录路径、渲染状态、尺寸和缩放比例。
- 非 `text_only` 页面还会根据 `image`、`table`、`caption`、`formula` 等 DocumentIR 块生成 `output/vision_crops/` 区域裁剪图，并在 `region_crops` 中记录块 ID、块类型、bbox、裁剪路径和尺寸。
- `output/ocr_tasks.json` 会把区域裁剪或页面预览进一步转成 `pending_engine` 任务；没有可视证据的任务会标记为 `blocked_missing_visual_evidence`，避免静默丢失 OCR 候选。
- 当前已完成本地路由、页面预览、候选区域裁剪、OCR 调度任务、tesseract CLI 本地 OCR 执行入口、OCR 结果文件入口、普通表格 OCR 文本本地结构化推断、文本级合并单元格候选推断、OCR 结果回写报告、OCR 候选 QA、OCR 候选晋级报告、晋级 DocumentIR 副本，以及通过门禁的表格/公式 OCR canonical 结构晋级；仍未接入 VLM。后续 OCR 适配器应输出 `ocr-results-v1` payload，由 `output/ocr_results.json` 固化后交给回写门禁追加为候选文本，再经 QA 和晋级门禁进入结构分段。

### 输出形态

**适用范围**：`exporters/`、Markdown 合并、双语 HTML 和 PDF 导出。

**规则**：
1. Markdown 继续作为轻量产物，但复杂论文不能只依赖纯 Markdown。
2. 双语 HTML 应优先保留结构、页码、块边界、图表占位和术语 hover。
3. PDF 输出应由结构化译文重新排版生成，不宜一开始追求完全回填原 PDF 坐标。
4. 表格、注释、人名、图注和引用在 HTML/PDF 中必须有清晰视觉层级。

**原因**：当前纯 Markdown 在表格、注释和图文混排场景下可读性不足，输出层必须承接结构识别的成果。

**验收**：
- 同一任务至少能输出结构化 Markdown、阅读版 HTML 和结构化 PDF 译文。
- PDF 输出原型能保留标题、段落、表格、QA 标记和基础页码关系；复杂图表、脚注和原版式关系进入后续验收。

## 推荐演进模块

后续新增模块建议按以下边界落地：

```text
pdf_translate/
  extractors/       # PDF/OCR/版面结构抽取，输出 DocumentIR
  chunkers/         # 页级、段落级、章节级、结构感知分块
  translators/      # 翻译服务适配与能力注册
  memory/           # 术语库、翻译记忆、摘要、冲突处理
  qa/               # 数字/公式/引用/术语/表格不变量检查
  exporters/        # Markdown、双语 HTML、结构化 PDF、zip 输出
  jobs/             # 队列、状态、取消、恢复、限流
  server/           # FastAPI 路由层
```

迁移不需要一次完成。新能力优先放进新目录，旧代码在有测试保护后逐步搬迁。

## 安全边界

- 生产环境必须设置管理员引导密码，不应依赖默认值。
- 对公网服务应显式配置 `PDF_TRANSLATE_CORS_ORIGINS`。
- `pdf_translate.server.security_preflight` 负责生产安全预检，CLI `pdf-translate security-check` 与管理端 `/api/admin/security/preflight` 必须复用同一份报告口径。
- JWT 必须带 `exp` 过期声明，登录有效期由 `PDF_TRANSLATE_JWT_TTL_MINUTES` 控制；服务端解码时必须拒绝过期 token 和缺少 `exp` 的历史 token。
- 当 `PDF_TRANSLATE_ENV` / `PDF_TRANSLATE_DEPLOYMENT_MODE` 进入生产模式时，Web app 初始化必须调用生产安全门禁，存在高/中风险配置时直接阻断启动。
- 上传大小由 `PDF_TRANSLATE_MAX_UPLOAD_MB` 控制，默认 120MB；Web 上传接口与安全预检必须使用同一套解析逻辑。上传保存必须流式读取，先校验 `%PDF-` 魔数，再按块写入任务目录，超限或非法文件不得留下半截输入文件。
- 管理端保存的 API Key 落在本地 SQLite；设置快照只返回 `secret_fields` 配置状态，不得回传密钥明文。设置 `PDF_TRANSLATE_SECRET_KEY` 或 `PDF_TRANSLATE_SECRET_KEY_FILE` 后，新保存的 API Key 必须以 `enc:v1:` 前缀密文落盘；旧明文值保持只读兼容，并由安全预检提示管理员重新保存加密。
- 用户临时 API Key 应只用于本次任务，不应默认落盘。
- 上传文件、输出文件和 zip 打包路径必须始终限制在任务工作目录内。zip 资料包遍历必须跳过 symlink，并确认候选文件 `resolve()` 后仍位于任务目录下。
