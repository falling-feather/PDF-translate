## 项目概览

本项目是面向英文论文 PDF 的本地可部署翻译工作台。当前核心闭环是：

```text
上传或指定 PDF
  -> 拆分正文与参考文献
  -> 提取正文页文本与基础富元数据
  -> 生成 DocumentIR、结构分段 manifest 与 OCR/VLM 路由报告
  -> 以 1-3 页滑窗分块
  -> 可选译前巡视并写入术语草稿
  -> 调用翻译后端
  -> 更新 memory/ 记忆
  -> 合并 Markdown
  -> 生成译后 QA、局部修复计划、局部修复请求、执行结果与验证结果
  -> 汇总专利实验指标
  -> 生成双语 HTML 阅读版
  -> 导出链接索引与 zip 资料包
```

当前项目优势是轻量、可恢复、术语与摘要记忆明确、支持串联/并联/精品翻译。当前短板是 PDF 结构层仍处在原型阶段，复杂表格、公式、多栏阅读顺序、跨页图注/脚注和扫描件 OCR 还没有形成稳定闭环。

## 当前技术栈

- 后端：Python 3.10+、FastAPI、Typer、SQLite、bcrypt、JWT。
- PDF：PyMuPDF，当前以 `page.get_text("text")` 为主。
- 翻译：OpenAI 兼容接口、DeepSeek、Ollama、DeepL、hybrid、echo。
- 前端：Vue 3、Vite，构建后由 FastAPI 静态托管。
- 任务数据：默认写入 `data/app.db` 和 `data/web_jobs/<job_id>/`。

| `docs/` | 正式项目文档 | `plans/` 只作为本地草稿，不作为协作入口 |

## 核心模块规则

### 翻译管线

**适用范围**：`pipeline.py`、未来 `extractors/`、`chunkers/`、`qa/`、`exporters/`。

**规则**：
1. 新增大功能时优先放到独立模块，避免继续扩大 `run_translate()`。
2. 管线中的每个阶段都要能产出可复查的中间文件或日志。
3. 翻译块输出必须保留 `chunk_id`、页码、后端、prompt 版本等元数据。
4. 串联模式要保护前文摘要、上一块译文尾部和段尾顺延协议。
5. 并联模式不得依赖串联顺延文件，避免跨线程污染上下文。

**原因**：后续要支持结构 IR、QA、局部重译和专利实验数据，必须让管线阶段可拆、可测、可追踪。

**验收**：
- 单个阶段失败时能定位到阶段、块和错误类型。
- 断点续译不会重复已完成块。
- `output/run_log.jsonl`、`state.json`、`chunks_manifest.json` 能解释一次运行。

### 后端注册

**适用范围**：`translators/factory.py`、`server/routes_web.py`、`settings_service.py`、前端管理端。

**规则**：
1. CLI、Web、管理员后台和用户自定义 API 应读取同一份后端能力表。
2. 能力表至少包含后端 id、展示名称、配置项、是否支持段尾顺延、是否支持并联、是否允许用户临时 key。
3. 不同后端的错误信息应映射为结构化错误码。

**原因**：当前翻译工厂支持多后端，但 Web 可选后端主要暴露 `echo/deepseek`，配置边界不够统一。

**验收**：
- 新增一个后端时不需要分别改 4-5 个位置。
- 前端展示、后端校验和实际构造器行为一致。

### 记忆与术语

**适用范围**：`memory_store.py`、译前巡视、精品翻译、未来术语确认台。

**规则**：
1. 术语应区分候选、已确认、已否决、冲突待审；当前新增术语默认写为 `candidate`。
2. 术语应保留来源、页码、章节、结构类型和更新时间。
3. 当前块只注入相关术语，避免全量术语挤占上下文窗口。
4. 模型草拟术语不能默认等同于人工确认术语。

**原因**：术语一致性是学术翻译质量的主要杠杆，也是专利方向中的动态记忆反馈证据。

**验收**：
- 同一英文多译、同一中文多源能进入 `pending_review.json` 或等价待审结构。
- 术语导入导出不破坏现有 `glossary.json`。

**当前落地**：
- `merge_glossary_terms_from_survey()` 新增术语时会写入 `status: candidate`。
- 同一英文术语出现不同中文译名时，不静默覆盖原术语，而是写入 `pending_review.json` 的 `glossary_conflict`。
- 多个英文术语共享同一中文译名时，会写入 `shared_translation_review`，供人工确认是否为同义术语。
- 译后 QA 会读取 `pending_review.json`，在相关块中报告 `glossary_translation_conflict`，修复计划会映射为 `review_glossary_conflict`。

### PDF 结构层

**适用范围**：未来 `extractors/`、`DocumentIR`、结构感知分块、QA。

**规则**：
1. 不直接把复杂 PDF 线性化为不可区分的普通文本。
2. 表格、图注、脚注、注释、人名、机构名、公式、标题、参考文献、页眉页脚应有类型边界。
3. 表格必须保留行列关系、表头、单位、脚注和显著性符号，不能被当作普通段落拼接。
4. 图注、脚注、页下注释和人名机构信息应能回到所属图表、段落或页面。
5. 分块不得切断表格、图注、公式环境和跨页未完段落。
6. 数字、公式编号、引用编号、单位和表格维度应作为结构不变量进入 QA。

**原因**：当前最大质量风险不是“模型不会翻译”，而是模型看到的输入结构已经被破坏。

**验收**：
- 对表格密集论文，输出能保留行列关系或至少给出风险标记。
- QA 报告能指出数字、引用、公式和表格维度异常。

**当前落地**：
- `pdf_translate.extractors.document_ir.extract_document_ir()` 会生成 `output/document_ir.json`。
- `pdf_translate.chunkers.structure.build_structure_chunks()` 会生成 `output/structure_chunks_manifest.json`，并在页面边界高风险时保护相邻页面不被目标字符数或普通页数阈值提前拆开。
- 表格块会在 `meta.table` 中记录行列、表头、数字 token、confidence 和 warning；结构分段会把表格优先渲染成 Markdown table。
- 图注、表注和脚注会在同页内按邻近结构块写入 `parent_id` 与 `meta.parent_relation`；紧跟表格的脚注会标记为 `footnote_for_table`，无法归属的内容会写入 `meta.parent_warning`。
- 文本块会用本地规则提取人名、机构、模型/数据集名和缩写候选，写入 `meta.entities`；这些结果只表示候选实体，不等同于人工确认术语。
- `pdf_translate.qa.structure.write_structure_qa()` 会生成 `output/structure_qa.json`，记录表格维度、续表风险、图注/表注/表格脚注/脚注关系、实体候选、孤立关系统计、页级风险、页面边界碎片和结构块统计。
- `pdf_translate.qa.table_reconstruction.write_table_reconstruction_report()` 会生成 `output/table_reconstruction.json`，按源表格输出单元格坐标、表头、行头、数字/单位/显著性 token、表注/脚注绑定和续表关系，作为表格保护式翻译、PDF 输出和专利实验的前置证据。
- `pdf_translate.qa.table_reconstruction.build_table_translation_hints()` 会把当前 chunk 覆盖的表格证据压缩为 `TranslationRequest.structure_hints`，OpenAI/DeepSeek 兼容后端会以「结构保护提示」注入模型消息，要求保留 Markdown 表格行列、表头/行头关系和锁定 token。
- `pdf_translate.qa.chunk_boundary.write_chunk_boundary_qa()` 会生成 `output/chunk_boundary_qa.json`，对照结构 QA 的 `page_boundary_fragments` 与当前实际 chunks，统计高风险页面边界是被保护、同块覆盖还是被切开。
- `pdf_translate.qa.chunk_boundary.write_chunk_strategy_comparison()` 会生成 `output/chunk_strategy_comparison.json`，同一份文档下同时评估固定页分段和结构分段的边界切开率、保护率和相对基线降幅。
- `pdf_translate.qa.translation.write_translation_qa()` 会生成 `output/qa_report.json` 和 `output/qa_report.md`，检查数字、引用、图表编号、Markdown 表格形状、DocumentIR 表格形状、表格单元格锁定 token、术语一致性、术语冲突、实体候选保留、重复段落和英文残留；术语一致性检查读取当前任务 `memory/glossary.json` 和 `memory/pending_review.json`，表格结构检查读取当前任务 DocumentIR 与 `table_reconstruction.json`。
- `pdf_translate.qa.repair.write_repair_plan()` 会生成 `output/repair_plan.json` 和 `output/repair_plan.md`，把 QA 问题映射成块级、段落级、表格级、表格单元格级、实体级或术语级修复建议，术语缺失会映射为带术语锁定要求的块级重译，实体缺失会映射为带实体锁定要求的块级重译，表格单元格 token 缺失会映射为 `repair_table_cell_tokens`，术语冲突会映射为人工术语复核。
- `pdf_translate.qa.repair.write_repair_requests()` 会生成 `output/repair_requests.json` 和 `output/repair_requests.md`，把修复计划转成可交给翻译后端或人工复核的请求信封，包含源文片段、当前译文片段、锁定 token、QA 证据、局部修复指令和后端 payload。
- `pdf_translate.qa.repair.write_repair_results()` 会生成 `output/repair_results.json` 和 `output/repair_results.md`；默认记录每条请求因执行开关关闭而跳过，CLI 显式传入 `--execute-repairs` 后才会调用翻译后端，并把修复片段写入 `output/repairs/`，不直接覆盖原译文 chunk。
- `pdf_translate.qa.repair.write_repair_validation()` 会生成 `output/repair_validation.json` 和 `output/repair_validation.md`，对成功生成的候选修复片段做本地不变量验证，当前重点检查锁定 token 保留与表格 Markdown 形状，不自动合并译文。
- `pdf_translate.qa.repair.write_repair_merge()` 会生成 `output/repair_merge.json`、`output/repair_merge.md`、`output/repaired_chunks/` 和 `output/repaired_full.md`；仅把验证通过且可安全定位的候选修复写入独立修复副本，当前支持 chunk 级替换和首个 Markdown 表格替换，不覆盖原始 `output/chunks/`。
- 管线会对 `output/repaired_chunks/` 再运行一轮译后 QA，生成 `output/repair_merge_qa.json` 和 `output/repair_merge_qa.md`，用于观察合并后 issue 数和表格 token 异常是否下降。
- `pdf_translate.qa.metrics.write_experiment_metrics()` 会生成 `output/experiment_metrics.json`，汇总结构 QA、表格重建、OCR/VLM 路由、译后 QA、修复计划、修复请求、执行结果、验证结果、合并结果和合并后 QA 中的页数、表格、单元格、续表、实体、页面边界、OCR 候选页、QA issue、修复项、可执行修复请求、已执行修复请求、表格锁定 token 缺失、修复验证通过率、修复合并应用率、合并后 issue 降幅和比例指标，作为专利原型实验的机器可读证据入口。
- `PageIR.meta` 会记录文本字符数、文本区域比例、图像区域比例和块类型计数，供 OCR/VLM 路由、QA 和后续输出层复用。
- `pdf_translate.vision.routing.write_vision_route()` 会生成 `output/vision_route.json`，按页面给出 `text_only`、`local_ocr`、`vlm_review` 或 `skip_blank` 建议和触发原因。
- CLI 可用 `--chunk-strategy structure` 试用结构分段翻译路径；默认仍为 `page`，避免直接改变 Web 用户路径。

**QA 边界**：
- 当前译后 QA 是纯规则检查，不调用模型，不直接修改译文。
- QA JSON 面向实验统计，QA Markdown 面向导师评审和人工复核。
- 结构边界检测会在翻译前检测相邻页面首尾块的断句、续表和接续风险，QA 输出 `page_boundary_fragments` 与 `table_continuations`，结构分段清单输出 `boundary_fragment_ids`，分段边界 QA 输出 `split_boundary_count`、`protected_boundary_count` 和 `high_risk_split_count`，策略对比输出 `active_split_reduction_vs_baseline`。
- 关系归属检测会在 `relationships` 中记录 caption/footnote 的父块、关系类型和孤立 warning，用于后续图文重排、PDF 输出和结构保持率统计。
- 表格重建报告会在翻译前保留表格单元格上下文，并按 chunk 注入结构保护提示；译后 QA 会把锁定 token 缺失定位到表格单元格坐标，但当前仍不自动修复合并单元格或真正执行单元格级重译。
- 实体候选检测会在 `entities` 中记录块 ID、页码、实体类型、文本、置信度和规则来源，用于后续术语确认台、实体一致性 QA 和专利实验统计。
- 译后实体 QA 会检查候选实体是否在译文中保留；当前只报告缺失候选，不判断中文译名是否正确。
- 译后表格 QA 会优先用 Markdown 表格形状比对；当源 chunk 不是 Markdown 表格但 DocumentIR 中存在表格块时，会按页码或结构 chunk 的 `block_ids` 读取源表格行列数，发现译文没有保留 Markdown 表格或行列不一致时输出 `table_shape_mismatch`；当目标表格对应单元格缺失数字、单位或显著性符号等锁定 token 时输出 `table_cell_token_mismatch`。
- 修复计划会读取 QA JSON 中的异常块，给出优先级、修复范围、执行器和证据；修复请求会继续读取源 chunk 与当前译文，生成可由后端执行的局部修复 payload；修复结果只记录执行状态与候选修复片段；修复验证只判断候选片段是否满足本地不变量；修复合并只写独立副本和合并报告，不覆盖正式译文。
- 实验指标不会替代明细文件；它只聚合可复现实验所需的摘要、比例和证据文件路径，后续批量样本对比应读取该文件并回链到结构 QA、分段边界 QA、译后 QA、修复计划、修复请求、修复结果、修复验证、修复合并和合并后 QA 等明细 JSON。
- 术语一致性 QA 目前可检查源文出现的英文术语是否使用术语库指定中文译名，并能暴露待审术语冲突；候选确认流程、置信度和跨章节作用域仍待完善。
- 当前修复执行不会自动合并回正式译文；后续自动修复器应消费 `repair_results.json`、`repair_validation.json`、`repair_merge.json`、`repair_merge_qa.json` 与 `output/repairs/`，在人工确认或策略通过后再替换正式译文，并记录失败回退。

**阅读输出当前落地**：
- `pdf_translate.exporters.bilingual_html.write_bilingual_html()` 会生成 `output/bilingual.html`。
- 双语 HTML 按 chunk 展示原文与译文，并在同页展示 QA 问题和局部修复建议。
- 当前 HTML 是评审/对照阅读产物，不追求 PDF 原坐标回填；后续 PDF 输出应从结构化块和修复后译文重新排版生成。

### 图像与 OCR

**适用范围**：未来 `extractors/`、OCR 适配器、页面截图、VLM 路由、图文 QA。

**规则**：
1. 不能只依赖“本页文本数量低于平均值”判断是否需要图像模型；该规则只能作为低成本启发。
2. 页面应保留文本层、图片区域、页面截图、图注和 OCR 置信度等信息。
3. 扫描页、图片型表格、公式密集页和图注密集页应进入风险列表。
4. 本地 OCR 优先服务于结构识别和图文边界恢复，不把图片翻译作为第一目标。

**原因**：图像能力的价值在于避免漏掉扫描页、图表和图注，而不是简单提高模型调用次数。

**验收**：
- 低文本页能区分扫描页、图片页、公式页和普通空白页。
- 开启 OCR/VLM 后，产物能说明哪些页面调用了图像处理，原因是什么。

**当前落地**：
- 管线会生成 `output/vision_route.json`，基于文本字符数、文本区域比例、图像数量、图像区域比例、图注/表格/公式块和页级 warning 给出路由建议。
- 当前仅完成本地路由与证据报告，不直接调用 OCR 引擎、页面截图或 VLM；后续 OCR 适配器应把识别结果回写到 DocumentIR，而不是生成孤立文本。

### 输出形态

**适用范围**：`exporters/`、Markdown 合并、未来双语 HTML 和 PDF 导出。

**规则**：
1. Markdown 继续作为轻量产物，但复杂论文不能只依赖纯 Markdown。
2. 双语 HTML 应优先保留结构、页码、块边界、图表占位和术语 hover。
3. PDF 输出应由结构化译文重新排版生成，不宜一开始追求完全回填原 PDF 坐标。
4. 表格、注释、人名、图注和引用在 HTML/PDF 中必须有清晰视觉层级。

**原因**：当前纯 Markdown 在表格、注释和图文混排场景下可读性不足，输出层必须承接结构识别的成果。

**验收**：
- 同一任务至少能输出结构化 Markdown 和阅读版 HTML。
- PDF 输出原型能保留标题、段落、表格、图注和页码关系。

## 推荐演进模块

后续新增模块建议按以下边界落地：

```text
pdf_translate/
  extractors/       # PDF/OCR/版面结构抽取，输出 DocumentIR
  chunkers/         # 页级、段落级、章节级、结构感知分块
  translators/      # 翻译服务适配与能力注册
  memory/           # 术语库、翻译记忆、摘要、冲突处理
  qa/               # 数字/公式/引用/术语/表格不变量检查
  exporters/        # Markdown、双语 HTML、zip、未来 PDF 输出
  jobs/             # 队列、状态、取消、恢复、限流
  server/           # FastAPI 路由层
```

迁移不需要一次完成。新能力优先放进新目录，旧代码在有测试保护后逐步搬迁。

## 安全边界

- 生产环境必须设置管理员引导密码，不应依赖默认值。
- 对公网服务应显式配置 `PDF_TRANSLATE_CORS_ORIGINS`。
- 管理端保存的 API Key 当前落在本地 SQLite；面向多人服务时需要明确提示或接入加密/密钥管理。
- 用户临时 API Key 应只用于本次任务，不应默认落盘。
- 上传文件、输出文件和 zip 打包路径必须始终限制在任务工作目录内。
