## 重要说明

本文是工程与方案梳理，不等同于专利法律意见。真正申请前，需要专利代理师围绕新颖性、创造性、实用性做检索、交底书整理和权利要求撰写。

## 基本判断

如果只把项目描述为“PDF 翻译工具”“AI 文献翻译方法”“使用术语表的翻译系统”或“保留格式的文档翻译方法”，很容易落入已有技术。更适合收束为一个具体技术方案：

> 针对学术 PDF 长文翻译中的上下文压缩、术语漂移、结构错乱和表格线性化问题，构建一种结构感知的层级式翻译控制方法：先将文档解析为带类型与边界的中间结构，再按结构原子和上下文预算生成翻译块，翻译过程中动态更新术语、摘要和衔接记忆，并通过结构不变量进行质量检测与局部重译修复。

当前更推荐把“算法级翻译流程”作为专利主线，而不是把“完全不借助 AI 的本地翻译工具”作为主线。OCR、图像识别、本地模型和 PDF 输出应作为支撑模块，服务于结构识别、低依赖运行和结果可读性。

## 当前核心问题

| 问题 | 当前表现 | 专利化切入点 |
| --- | --- | --- |
| 表格线性化 | 表格文字被抽成普通文本，行列、表头、单位和脚注丢失 | 表格结构保护、单元格上下文构造、表格维度 QA、单元格锁定 token QA |
| 注释/人名/图注归属丢失 | 脚注、图注、人名机构和正文混在一起 | 基于结构类型和父子关系的内容归位 |
| 图像与扫描页处理不足 | 仅靠文本数量估计是否需要图像模型，容易误判 | 本地 OCR + 页面截图 + 图像区域 + VLM 路由 |
| 分页切块造成碎片 | 页末半句、跨页表格、跨页图注被拆开 | 结构单元分段和跨块顺延记忆 |
| Markdown 可读性不足 | 复杂表格、脚注、图文混排在 Markdown 中表现差 | 结构化 Markdown、双语 HTML、PDF 译文生成 |

## 现有技术拥挤区

以下方向不宜单独作为核心创新点：

- 机器翻译结合术语表或翻译记忆。
- 普通上下文感知翻译记忆。
- 文档翻译后保留样式或原位置回填。
- 单纯把表格单元格拿出来翻译。
- 单纯保留 PDF 原排版、公式和图表。

这些能力可以作为系统组成部分，但不应作为唯一创新点。

## 建议核心名称

- 一种面向学术 PDF 的结构感知层级式机器翻译方法及系统。
- 一种基于结构中间表示和动态记忆反馈的学术 PDF 分层翻译方法。
- 一种基于结构不变量检测与局部修复的学术文档翻译方法。
- 一种基于本地结构识别与结构单元分段的学术 PDF 翻译排版方法。

## 面向专利的创新点汇总

当前更值得主打的创新点不是“接入某个大模型”，而是把论文翻译拆成一套可审计、可替换、可度量的流程算法：

1. **结构感知分段算法**：从按页切块改为按标题、段落、表格、图注、脚注和续表关系组织 chunk，减少页面交接碎片。
2. **表格结构保护与约束注入**：在翻译前识别表格行列、表头、单位、脚注、续表和疑似合并单元格，把这些约束压缩进翻译提示，并在译后用结构不变量 QA 检查。
3. **跨页父子关系归位**：为图注、表注、脚注和续表建立同页优先、一页跨页回退的父子关系，解决“文字属于哪里”的问题。
4. **OCR/VLM 混合路由闭环**：用页面文本量、图像区域、页面预览、区域裁剪、OCR 任务和候选晋级门禁决定是否进入本地 OCR 或后续 VLM，而不是单纯按文字数量触发。
5. **结构化 OCR 结果 canonical 晋级**：OCR 不只补充文本，而是把 OCR 返回或本地推断出的 `structured_cells`、`cell_bboxes` 和 `merged_cell_candidates` 经门禁晋级为 `block.meta.table`，把 `formula_latex` / `formula_tokens` / `equation_labels` 晋级为 `block.meta.formula`，让表格和公式作为结构事实进入后续分段、翻译和证据统计。
6. **动态术语与记忆控制**：把译前巡视、候选术语、冲突待审、chunk 摘要和译后术语 QA 串成闭环，为长文一致性提供可追踪依据。
7. **结构不变量驱动的局部修复**：把数字、引用、表格维度、锁定 token、实体和术语问题转成局部修复请求，再用本地门禁验证和独立合并副本降低全文重译成本。
8. **候选证据分级、人工确认审计与发布副本**：对合并单元格这类高风险结构，不直接把算法候选写成最终结构，而是生成候选、视觉/bbox 证据等级、默认复核建议和人工决策占位，支持 Web 轻量逐项写回，并在无阻断项后生成独立确认结构副本，形成“候选 -> 证据 -> 人审 -> 发布副本”的可追溯确认链。
9. **多形态阅读输出与证据归档**：同时生成结构化 Markdown、双语 HTML、结构化 PDF 译文、QA 报告、实验指标和资料包，便于导师评审、人工评分和专利交底。

本轮已把 OCR/VLM 混合路由继续细化为“表格/公式 OCR 结构化任务契约 + 普通表格 OCR 文本本地结构化与合并候选推断 + 结构化结果透传 + 本地结构门禁 + 结构化 JSON 执行器入口 + canonical 结构晋级”：表格区域不是普通 OCR 文本任务，而是携带源表格行列、表头、数字 token、locked token、bbox、子目标和可选结构返回字段的 OCR 任务；公式区域也携带公式上下文、锁定 token 和可选 LaTeX/token/编号结果契约。真实 OCR 或本地包装器一旦通过 stdout 返回 `ocr-results-v1`、结果列表或单条 JSON 结果，执行器会按 `task_id` 选取结果，回写、候选 QA、晋级与实验指标会继续保留 `structured_cells`、`cell_bboxes`、`merged_cell_candidates`、`formula_latex`、`formula_tokens` 等字段；如果表格 OCR 只给出 Markdown 管道表或列数稳定的普通文本，回写阶段也能本地推断 `structured_cells`、估算 `cell_bboxes`，并根据短行或空单元格生成疑似合并单元格候选。随后系统用行列坐标、行列数、bbox 覆盖、公式编号和 locked token 做本地可信门禁。通过门禁的表格/公式候选会进一步写入晋级 IR 的 `meta.table` / `meta.formula`，使结构分段和后续实验指标消费的是结构事实，而不是松散 OCR 文本。这可以作为“本地结构解析与 OCR 执行器解耦、再经门禁晋级为结构事实”的专利证据点。

本轮补充：OCR 晋级后的 `merged_cell_candidates` 已进一步接入表格重建与结构提示链路。系统会把 canonical `block.meta.table` 中的旧式或新式候选字段规范化为统一坐标、span 和覆盖单元格，再与本地 ragged 行/空单元格候选按几何范围合并去重，最终进入 `table_reconstruction.json`、`structure_hints_manifest.json` 和翻译请求的 `structure_hints`。这让“本地 OCR 文本 -> 结构化单元格 -> 疑似合并候选 -> canonical 表格 -> 翻译控制提示”的链路更完整，也更适合作为表格保护式翻译流程的专利证据点。

本轮补充：合并候选现在增加了证据分级防误用机制。系统会把候选本身、视觉/bbox 证据和人工确认状态拆成不同字段：文本启发式和区域均分 bbox 只能作为弱证据，外部 span bbox 可作为视觉支持证据，只有未来人工确认才可进入 `human_confirmed`。这使专利主线不只是“发现疑似合并”，而是形成“候选生成 -> 证据分级 -> 翻译保护 -> 人工确认/正式替换”的可审计流程。

本轮补充：合并候选人工确认审计产物已经落地。管线会在 OCR 晋级后重新生成表格重建证据，再输出 `table_merged_cell_review.json/md`，把每个合并候选整理为 `review_id`、候选状态、视觉/bbox 证据、默认决策和人工决策占位，并把统计结果写入 `experiment_metrics.json` 与资料包。该能力适合作为专利交底里的“高风险结构候选不直接生效，而是经证据分级和人工确认审计后进入正式替换”的流程证据。

本轮补充：合并候选人工确认现在已形成 Web 轻量闭环。用户端和管理端可下载 `table_merged_cell_review.md`，读取 `table_merged_cell_review.json`，并在列表式审核面板中按单条 `review_id` 写回 `confirm`、`reject`、`needs_revision` 或 `clear`；任务状态会同步统计待确认、已复核、已确认、已拒绝和需修改数量。该闭环仍不直接覆盖 `table_reconstruction.json`，适合在专利交底中表述为“候选结构先经证据分级和人审写回，再进入后续正式替换策略”的安全流程。

本轮补充：合并候选人工确认已进一步形成“发布副本”闭环。系统默认生成 `table_structure_publish.json/md` 记录待确认或阻断状态；只有人工确认清单没有待复核项时，Web 显式确认才会生成 `table_reconstruction_confirmed.json`。该副本不覆盖原始 `table_reconstruction.json`，而是作为后续 PDF 重排、单元格级修复和正式替换策略的输入；若审核项重新变为待复核或需修改，旧副本会失效删除。这可作为专利交底中的“高风险结构候选经人审门禁后形成独立结构事实副本”的流程证据。

这些点组合起来可以表述为“结构识别 -> 风险约束 -> 翻译控制 -> 本地 QA -> 局部修复 -> 可读输出 -> 实验度量”的算法链条。单点功能可能容易撞车，但完整闭环更像一个可申请的系统方法。

## 建议技术流程

```text
输入 PDF
  -> 本地结构解析与 OCR 风险检测
  -> 生成轻量 DocumentIR
  -> 译前巡视：术语、章节、表格/公式/图注/扫描页风险
  -> 用户确认术语与策略
  -> 结构单元分段：段落、表格、公式、图注、脚注不可切分
  -> 分层翻译：正文、表格、图注、公式附近文本、注释分别处理
  -> 动态记忆更新：术语、实体、摘要、尾部衔接
  -> 结构不变量 QA：数字、公式、引用、表格维度、术语一致性
  -> 局部修复：重译高风险块、段落或单元格
  -> 输出结构化 Markdown、双语 HTML、PDF 译文、QA 报告和资料包
```

## 可形成权利要求的组合方向

### 方向 A：结构感知层级翻译控制

**核心组合**：PDF 结构解析、结构原子标注、自适应分块、块间摘要、顺延记忆、翻译后记忆回流。

**技术效果**：
- 降低长上下文压缩导致的漏译。
- 降低跨块指代断裂。
- 降低术语漂移。
- 保持任务可恢复、可局部重译。

### 方向 B：表格保护式学术翻译

**核心组合**：表格结构识别、单元格上下文构造、数字/单位/符号锁定、表格维度 QA、异常单元格局部重译。

**技术效果**：
- 避免表格被线性化后语义错乱。
- 保持行列关系、表头和单位上下文。
- 提高表格短文本翻译准确性。

### 方向 B2：注释、图注与实体归位

**核心组合**：脚注/页下注释检测、图注/表注归属、人名/机构/数据集/模型名实体识别、正文与注释分层输出。

**技术效果**：
- 减少注释混入正文造成的阅读断裂。
- 保留人名、机构和专名上下文。
- 提高 Markdown、HTML 和 PDF 输出的可读性。

### 方向 C：动态记忆反馈式文档翻译

**核心组合**：译前术语候选、用户确认或置信度标记、翻译中术语回流、按页码/章节/结构类型选择性注入、冲突检测。

**技术效果**：
- 越翻译越稳定。
- 减少专业术语前后不一致。
- 降低上下文窗口浪费。

### 方向 D：结构不变量驱动的局部修复

**核心组合**：从源 DocumentIR 提取不变量、译后比对、定位异常块、选择修复粒度、只重译局部。

**技术效果**：
- 降低修复成本。
- 提高译文完整性。
- 避免全文重译引入新不稳定。

### 方向 E：图像页与 OCR/VLM 路由

**核心组合**：文本层统计、页面截图、图片区域检测、OCR 置信度、图注识别、VLM 调用决策、OCR 结果回写门禁、OCR 候选 QA、OCR 候选晋级和结构分段输入副本。

**技术效果**：
- 避免扫描页和图片型表格漏译。
- 减少对图像模型的无效调用。
- 让图像处理结果回流到结构 IR，而不是只生成零散描述。

## 最建议优先落地的专利原型

优先做 “A + B + B2 + D”，并把 “E” 作为支撑能力的最小闭环：

```text
DocumentIR
  -> 表格保护
  -> 注释/图注/实体归位
  -> 图像/OCR 风险路由
  -> 结构单元分段
  -> 动态记忆注入
  -> 结构不变量 QA
  -> 局部修复
  -> 双语 HTML / PDF 译文输出
```

原因：

- 与现有项目距离最近，不需要推翻当前管线。
- 能直接回应固定页分块和表格错乱问题。
- 比单纯术语记忆更有技术形态。
- 更容易用实验数据证明效果。
- 能把本地 OCR 和 PDF 输出纳入算法流程，而不是另起一条孤立路线。

## 原型模块要求

### DocumentIR

建议最小结构：

```json
{
  "doc_id": "paper-id",
  "pages": [
    {
      "page_no": 1,
      "blocks": [
        {
          "block_id": "p1-b3",
          "type": "paragraph | heading | table | table_cell | caption | footnote | formula | reference | person | organization | header_footer | image",
          "bbox": [0, 0, 0, 0],
          "text": "...",
          "parent_id": null,
          "order": 3,
          "locked_tokens": ["(1)", "F1-score", "Table 2"],
          "meta": {
            "parent_relation": "caption_for_table | caption_for_figure | footnote_for_table | footnote_for_block",
            "parent_warning": "orphan_caption | orphan_footnote",
            "parent_page_no": 1,
            "parent_page_gap": 0,
            "cross_page_parent": false,
            "cross_page_parent_attempted": false,
            "entities": [
              {
                "type": "person | organization | model_or_dataset | acronym",
                "text": "BERT",
                "confidence": "low | medium | high",
                "source": "organization_suffix | model_dataset_pattern | citation_author_year | uppercase_acronym"
              }
            ]
          }
        }
      ]
    }
  ]
}
```

验收：同一份 IR 能支撑分块、翻译、QA、双语输出和局部修复。

### 表格保护

必须保留：

- 行列结构。
- 表头、行名、单位、脚注。
- 跨页续表关系。
- 数字、显著性符号、公式片段。
- 单元格上下文提示。
- 译后行列数和数字集合比对。

验收：表格密集论文的表格行列保持率、续表风险和表格脚注归属可统计，异常能定位到表格、续表边界或单元格。

### 结构感知分块

分块依据：

- 结构原子不可切分。
- 模型上下文预算。
- 章节和段落边界。
- 表格/公式/图注风险。
- 脚注、注释和人名机构上下文。
- 跨页续段和续表。
- 跨块顺延记忆。

验收：相比固定页滑窗，重复译文、半句切断、术语漂移和漏译风险下降。

### QA 与局部修复

优先做规则 QA：

- 数字集合不变量。
- 公式/变量名不变量。
- 引用编号不变量。
- 表格维度不变量。
- 图表编号不变量。
- 实体候选保留不变量。
- 英文残留比例。
- 术语一致性。
- 重复译文检测。

验收：QA 能输出 `qa_report.md`，并能给出重译块、段落或单元格建议。

## 申请前证据清单

建议准备测试集：

- 10 篇普通英文论文。
- 10 篇表格密集论文。
- 5 篇公式密集论文。
- 5 篇多栏/图注复杂论文。
- 若做 OCR，本地再准备 5 篇扫描版或低文本层 PDF。
- 5 篇含大量脚注、作者/机构、人名实体或页下注释的论文。

对比组：

- 当前纯文本固定页分块版。
- 增加术语记忆版。
- 增加结构感知分块版。
- 增加表格保护版。
- 增加图像/OCR 路由版。
- 增加结构化 HTML/PDF 输出版。
- 增加 QA 局部修复版。

指标：

- 每次任务的 `experiment_metrics.json`，用于回链结构 QA、表格重建、OCR/VLM 路由、OCR 候选 QA、OCR 候选晋级、译后 QA、局部修复计划、局部修复请求、局部修复结果、局部修复验证、局部修复合并、合并后 QA 和运行性能指标。
- 每次任务的 `run_metrics.json` / `run_log.jsonl`，用于记录阶段耗时、翻译请求次数、HTTP 尝试/重试次数、字符数、估算 token、吞吐和跳过块。
- 每次任务的 `vision_route.json`、`vision_pages/`、`vision_crops/`、`ocr_tasks.json`、`ocr_results.json`、`ocr_writeback.json`、`ocr_candidate_qa.json`、`ocr_candidate_promotion.json`、`document_ir_ocr.json` 与 `document_ir_promoted.json`，用于说明哪些页面进入 OCR/VLM 候选、原因是什么、对应页面预览和候选区域裁剪证据在哪里、后续本地 OCR/VLM 任务应如何执行、本地 OCR 是否尝试/成功/失败、实际 OCR 结果载荷是什么、OCR 候选文本是否通过门禁、以及哪些候选最终进入结构分段输入副本。
- 每次任务的 `cost_estimate.json`，用于记录成本画像命中情况、币种、单位价格、估算 token/字符/请求用量、可计费请求数和估算金额。
- 漏译率。
- 术语一致率。
- 表格行列保持率。
- 表格可重建率与单元格级数字/单位/显著性 token 覆盖。
- 合并单元格候选数、候选率、类型分布和原因分布。
- 表格形状异常率。
- 表格单元格 token 异常率与锁定 token 缺失率。
- 图注/脚注归位准确率。
- 跨页父子关系成功率。
- 图像/扫描页覆盖率。
- 页面交接碎片率。
- 页面边界切开率 / 保护率。
- 结构分段相对固定页分段的边界切开降幅。
- 预算越界块率。
- 结构关系保护次数。
- 数字/公式/引用保留率。
- 英文残留比例。
- 人工可读性评分：按 Markdown、HTML、PDF、表格、图文脚注、术语一致性和结构连贯性分别记录 1-5 分，并标记是否纳入专利证据。
- 单篇耗时、API 调用次数、HTTP 重试次数、估算 token、吞吐和估算成本；当前已具备耗时/字符/token 估算、HTTP 尝试回写与可配置价格画像，失败尝试 token 成本仍需结合服务商账单校准。
