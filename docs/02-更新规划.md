## 背景与问题

当前工具已经完成 PDF 上传、正文/参考文献拆分、分块翻译、记忆注入、串联/并联、精品翻译、多用户 Web 工作台和资料包下载。后续如果目标是充分做好并准备专利，重点不应只是“多接几个模型”，而应把工具做成可验证的学术翻译系统：

- 解除对外部 API 的全程依赖，支持本地 OCR/结构解析/本地模型的可选闭环。
- 优化固定页数分块，减少段落、表格、公式和图注被切断的问题。
- 引入结构中间表示，让翻译、QA、局部修复基于同一份结构事实。
- 改善表格、注释、人名、图注和脚注在纯文本抽取后失去排版关系的问题。
- 建立图像、扫描页和低文本页的本地 OCR/VLM 路由，替代单一文本数量启发。
- 将输出从纯 Markdown 扩展为结构化 Markdown、双语 HTML 和可评估的 PDF 译文。
- 把“精品翻译”从更贵模型升级为译前计划、术语确认、结构风险识别和 QA 的流程。
- 形成可量化实验数据，为专利交底和论文/软著材料准备证据。

## 阶段目标

1. 稳定当前 MVP，补齐测试、错误码、后端注册表和任务状态边界。
2. 做出结构化阅读体验：表格保护、注释归位、人名/机构识别、中英对照输出和 QA 报告。
3. 建立结构感知算法原型：DocumentIR、结构单元分段、动态记忆、结构不变量 QA。
4. 增强本地化能力：OCR/版面解析可本地运行，图像页可路由到 OCR/VLM，翻译后端支持本地模型或私有接口。
5. 建立专利证据链：测试集、对比实验、质量指标、成本与耗时日志。

## 当前落地进度

2026-06-29 已完成结构层基础原型：

- 新增轻量 `DocumentIR`，可从 PyMuPDF 本地抽取页面、结构块、bbox、块类型、锁定 token 和页级风险。
- 新增实验性结构分段器，能够保留块来源、块类型统计、页级 warnings、受保护页面边界，并输出 `structure_chunks_manifest.json`。
- 表格块已具备启发式行列、表头、数字 token、confidence 和 warning；结构分段时会优先渲染为 Markdown table。
- 图注、表注和脚注已具备同页基础归属：`DocumentIR` 会写入 `parent_id` / `parent_relation`，紧跟表格的脚注会标记为 `footnote_for_table`，`structure_qa.json` 会统计 linked/orphan 与 `table_footnote_count` 并输出 `relationships` 明细。
- 跨页相邻表格已进入续表风险统计：`structure_qa.json` 会输出 `table_continuations`，并在表格条目中记录 `continued_from_block_id` / `continued_to_block_id`。
- `structure_qa.json` 已增加页面边界碎片检测，可记录相邻页面首尾块的断句、续表和接续风险，并统计 `page_boundary_fragment_rate`；结构分段器会消费这些边界风险，基础保护高风险相邻页面。
- `PageIR.meta` 已记录文本字符数、文本区域比例、图像区域比例和块类型计数。
- 管线每次翻译会写出 `output/document_ir.json`、`output/structure_chunks_manifest.json`、`output/structure_qa.json`、`output/vision_route.json`、`output/qa_report.json`、`output/qa_report.md`、`output/repair_plan.json`、`output/repair_plan.md` 和 `output/bilingual.html`。
- 新增页面级 OCR/VLM 路由基础报告，可按 `text_only`、`local_ocr`、`vlm_review`、`skip_blank` 给出处理建议、风险分数和触发原因。
- 新增译后规则 QA 基础报告，可检查数字、引用、图表编号、表格 Markdown 形状、术语一致性、重复段落和英文残留。
- 新增 QA 驱动局部修复计划，可把 QA 问题映射为块级、段落级或表格级修复项，记录优先级、修复动作、执行器和证据；术语缺失会进入术语锁定重译建议。
- 新增术语冲突待审基础：巡视合并术语时会把同一英文多译和同一中文多源写入 `pending_review.json`，译后 QA 会将相关块标记为 `glossary_translation_conflict`。
- 新增双语 HTML 阅读版，按块展示原文、译文、QA 问题和局部修复建议。
- CLI 增加 `--chunk-strategy structure`，可试用结构分段翻译路径；默认仍为 `page`。

仍未完成：

- 高精度表格行列重建、续表合并重建、表格脚注单元格级绑定、跨页/多栏图注脚注归位、人名机构实体归位、真实 OCR 引擎/VLM 调用、术语确认台/置信度/章节作用域管理、PDF 译文输出、自动局部重译，以及更细粒度的段落/表格/图注级边界合并策略仍待继续实现。

## 几大核心方向

### 方向一：算法级翻译流程

这是最适合作为专利主线的方向。核心不是某个模型，而是流程控制算法：

```text
本地结构识别
  -> 结构单元分段
  -> 译前术语与风险计划
  -> 动态记忆注入
  -> 分层翻译
  -> 结构不变量 QA
  -> 局部修复
```

验收重点是每一步都有中间结果、指标和可复现对比。

### 方向二：表格、注释、人名和图注处理

当前纯文本抽取会把表格行列、脚注、图注、人名机构信息打散，导致 Markdown 可读性差。该方向要解决“文本属于哪里”的问题：

- 表格保留行列、表头、单位、脚注、显著性符号。
- 图注和表注保留与图表编号的关系。
- 人名、机构名、数据集名、模型名进入实体识别和术语记忆。
- 页脚注释、括注、参考引用不能混入正文段落。

验收重点是结构保持率和阅读产物可读性。

### 方向三：图像、扫描页和 OCR/VLM 路由

现有“按每页文字数量低于平均值来预估是否调用图像模型”的方法只能作为粗略启发。后续应综合：

- 页面文本字符数和文本区域面积。
- 图片区域数量、面积和位置。
- 页面截图 OCR 置信度。
- 是否存在图注、表格截图、公式密集区域。
- 是否需要本地 OCR、是否需要 VLM、是否只做图像占位。

验收重点是低文本页不漏处理，且不会对普通文本页过度调用昂贵模型。

### 方向四：分段替代分页

分页切块会制造页面交接碎片，尤其是跨页段落、表格续页、公式推导和图注。后续分块策略应从“页”转向“结构单元”：

- 段落、标题和章节优先。
- 表格、公式、图注不可切分。
- 跨页未完段落用顺延机制接续。
- 根据上下文预算动态调整块大小。

验收重点是半句切断率、重复译文率、漏译率和块间衔接评分。

### 方向五：输出形态从 Markdown 扩展到 HTML/PDF

Markdown 适合轻量资料包，但复杂论文需要更强排版表达：

- 结构化 Markdown：保留标题、表格、图注、脚注。
- 双语 HTML：适合对照阅读、术语 hover、块级定位、QA 标记。
- PDF 译文：适合导师评审和正式传播，但建议由结构化译文重新排版生成，不先追求完全原坐标回填。

## 本地化 OCR 与 API 依赖解除路线

本地化不是一次性替换所有模型 API，而是分层降低依赖：

1. **本地解析**：优先把 PDF 文本、表格、注释、图注、公式边界在本地抽取，外部 API 只负责必要翻译。
2. **本地 OCR**：对扫描页、低文本页、图片型表格使用本地 OCR，输出 OCR 置信度与风险标记。
3. **本地结构 QA**：数字、公式、引用、表格维度、图表编号等 QA 规则本地运行，不调用模型。
4. **本地排版输出**：HTML/PDF 生成尽量本地完成，减少把排版恢复交给模型。
5. **本地翻译后端**：Ollama/私有 OpenAI 兼容接口作为可选翻译端，先保证接口一致性。
6. **混合路由**：普通文本走本地/低成本模型，高风险表格、图像页或复杂段落再调用强模型。

验收重点不是“完全无 API”，而是可证明 OCR、结构解析、QA、排版输出已经本地化，翻译执行器可替换。

## 分段式翻译算法优化路线

当前固定页数滑窗简单可靠，但不能感知论文结构。建议分四步升级：

1. **段落边界保护**：先从纯文本层识别段落、标题、公式编号和引用编号，避免明显半句切块。
2. **跨页续段识别**：识别页面末尾未完句、表格续页、图注续页，交给顺延机制处理。
3. **结构原子分块**：引入 DocumentIR 后，把表格、图注、公式环境、注释和脚注作为不可切分对象。
4. **预算驱动分块**：按模型上下文预算、术语数量、前文摘要长度和结构风险动态决定块大小。

核心验收指标：

- 重复译文率下降。
- 块间断裂/漏译率下降。
- 术语一致率提升。
- 页面交接碎片率下降。
- 单块失败可恢复，且局部重译不会破坏全局合并。

## 依赖与风险

- OCR/版面解析会引入额外模型或原生依赖，Windows 部署复杂度会上升。
- 表格结构识别可能需要多工具组合，不能指望单一库覆盖所有论文。
- 本地大模型翻译质量和速度受显卡/内存限制，不能作为唯一默认方案。
- 专利方向需要新颖性检索，工程创新点不能直接等同于可授权权利要求。
- 如果过早追求 PDF 原排版回写，项目会被拖入重型排版系统，偏离当前轻量优势。

## 不做事项

- 本阶段不整体重写为 C++。
- 本阶段不优先做完整 PDF 原排版回写。
- 本阶段不把“精品翻译”仅定义为更贵模型。
- 本阶段不盲目接入大量翻译后端，先统一注册表和质量闭环。
- 本阶段不承诺完全离线翻译，先做到解析、OCR、QA 的本地化可选能力。

## 验收方式

- 自动测试：核心纯函数和管线边界必须有不依赖外部 API 的测试。
- 人工评审：准备普通论文、表格密集论文、公式密集论文、多栏复杂论文样本。
- 指标记录：漏译率、术语一致率、数字/公式/引用保留率、表格行列保持率、图像/扫描页覆盖率、页面交接碎片率、输出可读性、耗时、成本。
- 文档同步：完成任务后更新发布历史和测试指南。

## 待确认问题

- 本地 OCR 首选引擎是 PaddleOCR、Tesseract、RapidOCR，还是通过 MinerU/Marker/Docling 统一接入。
- 双语输出优先做 HTML，还是同时推进结构化 Markdown 与 PDF 译文。
- 术语确认台是否作为所有模式前置步骤，还是只绑定精品翻译。
- 专利原型的最小实验集规模和导师验收口径。
