## 结论摘要

市面和开源生态大致分为四类：

1. **重型 PDF 原排版翻译**：目标是保留原 PDF 布局、公式、图表，例如 PDFMathTranslate、BabelDOC。
2. **云端文档翻译服务**：目标是上传文档并直接翻译，例如 DeepL、Microsoft Translator Document Translation。
3. **浏览器/阅读器翻译插件**：目标是即开即读，适合网页和一般 PDF 阅读，但对论文结构、术语治理和实验证据支持有限。
4. **文档解析/OCR 底座**：目标是把 PDF、图片、Office 文档转成结构化文本或 Markdown，例如 Docling、MinerU、Marker、PyMuPDF4LLM。

本项目不宜和第一类正面竞争“完整 PDF 回写”。更合理的差异化是：

> 本地可部署、面向学术阅读、结构感知分块、动态术语记忆、对照输出、QA 局部修复、可沉淀实验数据。

进一步说，本项目应把“表格、注释、人名、图像和跨页碎片”的处理作为学术 PDF 场景差异化，而不是只追求通用文档翻译。

| --- | --- | --- | --- | --- |
| PDFMathTranslate | 开源科学 PDF 翻译 | 强调保留 PDF 布局、公式、图表，支持多种使用形态 | 说明“原排版 PDF 翻译”已有成熟路线，本项目应避开单纯保格式叙事 | 不宜把“保留原格式”作为唯一专利点 |

| Microsoft Translator Document Translation | 云端批量文档翻译 | 支持批量异步文档翻译和多格式处理 | 任务队列、批处理、状态管理可参考 | 与本项目轻量私有部署定位不同 |
| Docling | 文档转换/解析 | 面向 PDF、Office 等文档结构转换，包含布局与表格相关能力 | 可作为重解析适配器候选 | 引入依赖和部署复杂度 |
| MinerU | 文档解析/OCR | 面向 PDF 解析、OCR、公式、表格等结构化抽取 | 可作为本地 OCR/结构解析候选 | 资源占用和模型环境需要评估 |
| Marker | 文档转 Markdown/JSON | PDF、图片、Office 到 Markdown/JSON，覆盖 OCR 和表格等场景 | 适合作为本地结构抽取候选 | 输出质量需要用论文测试集验证 |
| PyMuPDF4LLM | PDF 到 Markdown 抽取 | 在 PyMuPDF 生态内生成 LLM/RAG 友好文本 | 作为当前 PyMuPDF 纯文本抽取的低成本增强 | 不等同于完整版面理解 |
| GROBID | 学术文档结构抽取 | 标题、作者、摘要、章节、参考文献等学术结构 | 适合后续论文元数据、章节和引用结构 | 对 PDF 文本层质量有要求 |

## 与本项目的定位差异

### 不做“又一个重型 PDF 排版翻译器”

完整 PDF 回写需要处理字体、坐标、换行、图层、公式、图片、版面重排和跨语言文本长度变化。该路线已有 PDFMathTranslate、BabelDOC 等项目，而且工程复杂度很高。

本项目应把第一阶段目标放在：

- 学术阅读可用。
- 术语一致。
- 上下文连续。
- 表格和公式不被破坏。
- 可对照、可审查、可局部修复。

### 不做单纯云端翻译包装

如果只是上传 PDF 后调用一个模型翻译，差异化较弱。项目更有价值的地方应是本地任务工作台、术语记忆、结构分块、QA 报告和可恢复资料包。

### 把 OCR 作为解除 API 依赖的一部分

本地化 OCR 的价值不是为了“翻译图片本身”，而是为了让扫描页、图注、表格和公式附近文本能进入结构化流程。OCR 应服务于 DocumentIR 和 QA，而不是成为孤立功能。

## 建议技术取舍

| 方向 | 建议 | 原因 |
| --- | --- | --- |
| PDF 抽取 | 先保留 PyMuPDF，增加 PyMuPDF4LLM/Marker/Docling/MinerU 适配层 | 降低迁移风险 |
| OCR | 先做风险检测和可选 OCR，不默认强制 | 避免拖慢普通文本 PDF |
| 表格 | 优先做表格结构保护和 QA | 表格错乱最影响学术论文理解，也最容易证明技术效果 |
| 注释与实体 | 将脚注、图注、人名、机构名从普通正文中分离 | 改善阅读体验，也能支撑术语记忆 |
| 原排版 PDF 输出 | 后置 | 当前差异化在阅读和质量控制，不在排版复刻 |
| 结构化 PDF 输出 | 可作为中期产物 | 从 IR 重新生成可读 PDF，比完全回写原 PDF 更现实 |
| 翻译后端 | 统一 registry 后再扩展 | 避免 CLI、Web、管理端能力不一致 |
| 本地模型 | 作为可选后端和混合路由 | 本地模型质量/速度差异大，不宜唯一默认 |
