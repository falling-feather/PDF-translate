## 当前方向总览

当前最合适的专利与产品路线，是把“算法级翻译流程”作为核心，把本地 OCR、图像识别、结构解析和输出排版作为支撑能力。翻译模型、传统机器翻译或本地模型都应被视为可替换执行器，项目真正要沉淀的是一套面向学术 PDF 的结构感知分层翻译流程。

| 方向 | 核心问题 | 建议目标 |
| --- | --- | --- |
| 算法级翻译流程 | 只调用模型翻译不够形成差异化 | 形成“结构识别 -> 分段 -> 记忆 -> 翻译 -> QA -> 局部修复”的可验证流程 |
| 表格与结构化内容 | 当前纯文本抽取会破坏表格、注释、人名、图注和脚注关系 | 引入结构中间表示，保留行列、注释归属、图表编号和专名上下文 |
| 图像、扫描页与 OCR | 仅靠文字数量估计是否调用图像模型不够稳定 | 建立本地 OCR、页面截图、图像区域、图注和 VLM 路由策略 |
| 分段替代分页 | 当前按页切块会造成页面交接碎片和跨页半句 | 以段落、章节、表格、公式、图注等结构单元决定块边界 |
| 输出形态升级 | 纯 Markdown 对复杂论文表格和图文排版不友好 | 提供阅读版双语 HTML、结构化 Markdown，并评估生成 PDF 译文 |
| 本地化与低 AI 依赖 | 完全不借助 AI 做高质量学术翻译成本过高 | 优先把 OCR、结构解析、QA、排版恢复本地化，翻译执行器保持可替换 |
| 专利证据链 | 没有实验指标会削弱交底说服力 | 记录漏译率、术语一致率、表格保持率、图像覆盖率、块间碎片率和修复成本 |

## 推荐主线

专利主线建议收束为：

> 一种基于本地结构识别、结构感知分段、动态记忆反馈与结构不变量校验的学术 PDF 分层翻译方法。

其中“本地文本识别与图像处理”不是孤立卖点，而是支撑结构识别和降低外部 API 依赖的关键模块。这样既保留算法层级的新意，又避免把项目压到“自研 OCR 或自研翻译模型”这种成本极高的路线上。

## 文档边界

- `README.md` 只保留项目简介、快速启动和文档入口。
- `SETUP_MANUAL.md` 负责安装、环境变量、API 配置和启动方式。
- `docs/01-开发者文档.md` 记录长期技术规则，不记录短期任务流水账。
- `docs/02-更新规划.md` 只记录未来仍需推进的任务，已完成内容迁入发布历史。
- `docs/03-专利优化方向.md` 是工程交底素材，不等同于法律意见；正式申请前需要专利代理师做新颖性检索与权利要求撰写。
- `plans/` 仅作为本机草稿来源，正式协作以 `docs/` 为准。

## 文档维护流程

1. 新功能开工前，在更新规划中写清目标、优先级和验收标准。
2. 新增长期技术规则时，同步更新开发者文档或新增 ADR。
3. 涉及用户流程、质量指标、专利证据的改动，同步更新测试与验收指南。
4. 功能完成后，把完成事实、验证方式和剩余风险写入发布历史。
5. README 只补入口链接，不堆积内部实现细节。
